# [ Understanding the Architecture of React Applications ]{#kobo.2.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h1_14 .chapterTitle}

[ React is the most ]{#kobo.3.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_f3348125 .index-entry index-entry="React"} [ popular JavaScript library in the world for building user interfaces. ]{#kobo.4.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ The job market is full of React-based opportunities. ]{#kobo.5.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ React has become so popular for multiple reasons: its small API, its massive ecosystem of pre-built components, and its flexibility. ]{#kobo.6.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Yet for the modern developer, that flexibility can also be a double-edged sword. ]{#kobo.7.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ This is because React doesn\'t provide a strong opinion on how to structure your application, so every team has to figure it out on their own. ]{#kobo.8.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ That leads to inconsistency across projects, architectural decisions made under deadline pressure without fully understanding the tradeoffs, and codebases that become increasingly difficult to maintain as they grow. ]{#kobo.9.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

# [ Why architecture matters ]{#kobo.29.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h1_15 .heading-1}

[ Every ]{#kobo.30.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_c188fd68 .index-entry index-entry="architecture:significance"} [ application has an architecture, even those where nobody thought about it. ]{#kobo.31.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ You might open a React project and find 47 components in a single ]{#kobo.32.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} ` `{.inlineCode}[` components/ `{.inlineCode}]{#kobo.33.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}` `{.inlineCode} [ folder, API calls scattered across random files, and state managed three different ways depending on who wrote the code. ]{#kobo.34.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ That\'s architecture. ]{#kobo.35.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ It just wasn\'t a deliberate one. ]{#kobo.36.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ The question isn\'t whether your application has architecture, but whether that architecture was chosen deliberately or happened by accident. ]{#kobo.37.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ When we talk about architecture, we\'re talking about the high-level decisions that shape how your application works. ]{#kobo.38.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ How is the code organized? ]{#kobo.39.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ How do different parts communicate? ]{#kobo.39.2 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ What patterns does the team follow? ]{#kobo.39.3 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ These decisions are difficult to change later because they affect everything built on top of them. ]{#kobo.39.4 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ In this chapter, we\'ll explore why architecture matters in the first place, look at the specific architectural challenges React presents, examine patterns that work and patterns that don\'t, and then plan the real application we\'ll build throughout the rest of the book. ]{#kobo.40.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ Architecture as foundation ]{#kobo.41.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_16 .heading-2}

[ When ]{#kobo.42.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_8306f98d .index-entry index-entry="architecture:as foundation"} [ you construct a building, you start with the foundation. ]{#kobo.43.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ You don\'t start with the walls and figure out the foundation later. ]{#kobo.44.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Software works the same way, but the consequences aren\'t as immediate. ]{#kobo.45.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ You can start building features without thinking about architecture, and things will work fine for a while. ]{#kobo.46.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ The problems show up later, when you need to change something. ]{#kobo.47.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ A solid architectural foundation makes your application resilient to change. ]{#kobo.48.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ And change is the only constant in software. ]{#kobo.49.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Requirements shift. ]{#kobo.50.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ New features get added. ]{#kobo.51.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Team members come and go. ]{#kobo.52.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ New ]{#kobo.53.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_77560704 .index-entry index-entry="architecture:as foundation"} [ technologies appear. ]{#kobo.54.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Business priorities pivot. ]{#kobo.55.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ An application built on a solid foundation can adapt to these changes. ]{#kobo.56.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ One built on a shaky foundation requires increasingly expensive workarounds until eventually the only option is the most expensive of all: a rewrite. ]{#kobo.57.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ Coordination and velocity ]{#kobo.58.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_17 .heading-2}

[ Good architecture ]{#kobo.59.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_5cec66df .index-entry index-entry="architecture:coordination and velocity"} [ makes it easier to work as a team. ]{#kobo.60.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When components have clear boundaries and responsibilities, developers can work on different parts of the application without stepping on each other\'s toes. ]{#kobo.61.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ You can split work across team members more effectively. ]{#kobo.62.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ You can make better estimates because you understand the scope of changes. ]{#kobo.63.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ This matters even more as teams grow. ]{#kobo.64.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ A solo developer can keep an entire poorly architected application in their head, but that doesn\'t scale. ]{#kobo.65.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When a second developer joins, they need to understand the patterns. ]{#kobo.66.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When a third joins, they need those patterns to be consistent. ]{#kobo.67.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ By the time you have five developers, inconsistent architecture creates constant friction. ]{#kobo.68.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ Bad architecture also affects AI-assisted development. ]{#kobo.69.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When you use tools like GitHub Copilot or Claude, they predict based not just on their training data but on your codebase patterns. ]{#kobo.70.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Consistent architecture means consistent suggestions. ]{#kobo.71.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Inconsistent architecture means the AI can\'t figure out which pattern to follow, and you spend more time correcting its suggestions than you save from using it. ]{#kobo.72.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ Speed through clarity ]{#kobo.73.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_18 .heading-2}

[ Counterintuitively, spending time on architecture upfront actually increases development speed. ]{#kobo.74.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When ]{#kobo.75.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_c92a3d2b .index-entry index-entry="architecture:speed"} [ architectural decisions are made, developers can focus on solving business problems instead of debating technical approaches for every feature. ]{#kobo.76.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ New features fit into existing patterns. ]{#kobo.77.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ New developers can become productive quickly because the structure is clear. ]{#kobo.78.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ The alternative is what we might call \" ]{#kobo.79.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} *[ architecture decision fatigue ]{#kobo.80.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}* [ .\" ]{#kobo.81.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ Every new feature requires deciding how it should work, where it should live, and how it should communicate with other parts of the application. ]{#kobo.81.2 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ These micro-decisions add up. ]{#kobo.82.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ They create inconsistency. ]{#kobo.83.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ They slow everything down. ]{#kobo.84.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ This is particularly important when working with LLMs for code generation. ]{#kobo.85.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ If you ask an AI to add a feature to a well-architected application, it can follow existing patterns. ]{#kobo.86.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ If you ask it to add a feature to an inconsistent codebase, it has to guess which pattern to follow, and those guesses are often wrong. ]{#kobo.87.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ Economics of good architecture ]{#kobo.88.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_19 .heading-2}

[ Software ]{#kobo.89.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_5d968887 .index-entry index-entry="architecture:economics"} [ projects cost money. ]{#kobo.90.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Most of that cost is people: their time, their effort, their attention. ]{#kobo.91.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Good architecture reduces costs by making those people more effective. ]{#kobo.92.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ It reduces the time spent debugging. ]{#kobo.93.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ It reduces the time spent searching for where things are. ]{#kobo.94.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ It reduces the time spent working around architectural limitations. ]{#kobo.95.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ Poor architecture has a compounding cost. ]{#kobo.96.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Every new feature takes longer because you have to work around existing problems. ]{#kobo.97.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Every bug is harder to fix because the boundaries between components are unclear. ]{#kobo.98.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Eventually, you reach a point where adding new features is prohibitively expensive and a rewrite becomes necessary. ]{#kobo.99.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Rewrites are expensive and risky. ]{#kobo.100.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ Good architecture also helps you make better business decisions. ]{#kobo.101.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When you understand your application\'s structure, you can estimate costs more accurately. ]{#kobo.102.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ You can decide which features to build and which to defer until later. ]{#kobo.103.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ Architecture and product quality ]{#kobo.104.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_20 .heading-2}

[ When ]{#kobo.105.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_136161dd .index-entry index-entry="architecture:product quality"} [ developers spend less time fighting the architecture, they have more time to focus on user needs. ]{#kobo.106.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ They can spend time on polish. ]{#kobo.107.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ They can think about edge cases. ]{#kobo.108.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ They can improve error handling. ]{#kobo.109.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ They can make the application more accessible. ]{#kobo.110.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ Poor architecture creates a vicious cycle. ]{#kobo.111.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Developers spend time fixing architectural problems. ]{#kobo.112.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Features ship with rough edges. ]{#kobo.113.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Users encounter bugs. ]{#kobo.114.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Developers spend more time on emergency fixes. ]{#kobo.115.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ There is never time to improve the architecture because there is always something urgent. ]{#kobo.116.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ Good architecture creates a happy cycle of improving quality. ]{#kobo.117.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Developers spend time building features. ]{#kobo.118.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Features ship with better quality. ]{#kobo.119.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Users are happier. ]{#kobo.120.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Fewer bugs come back. ]{#kobo.121.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ The team has time to keep the architecture healthy. ]{#kobo.122.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ The goal of any software is to solve real problems for real people. ]{#kobo.123.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Everything else is in service to that goal. ]{#kobo.124.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Architecture matters because it determines whether you can stay focused on that goal or get lost in technical debt. ]{#kobo.125.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

# [ The React architecture challenge ]{#kobo.126.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h1_21 .heading-1}

[ React is an ]{#kobo.127.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_52420d46 .index-entry index-entry="React architecture:challenge"} [ excellent tool for building user interfaces, but it intentionally doesn\'t solve certain problems. ]{#kobo.128.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Understanding what React does and doesn\'t do helps us understand the architectural decisions we need to make. ]{#kobo.129.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ How do we structure the project? ]{#kobo.147.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_22 .heading-2}

[ React doesn\'t have an opinion about file structure. ]{#kobo.148.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ You could: ]{#kobo.149.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- [ Put everything in one file (though you shouldn\'t) ]{#kobo.150.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}
- [ Have a flat folder of components ]{#kobo.151.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}
- [ Organize by technical role (all components in one folder, all hooks in another) ]{#kobo.152.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}
- [ Organize by feature ]{#kobo.153.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}
- [ Organize by domain concept ]{#kobo.154.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ Dan Abramov, once ]{#kobo.155.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_cf59fcc7 .index-entry index-entry="React architecture:challenge"} [ one of React\'s core team members, famously said: \" ]{#kobo.156.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} *[ Move files around until it feels right ]{#kobo.157.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}* [ .\" ]{#kobo.158.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ This is helpful, but for some it can be frustrating. ]{#kobo.158.2 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ It\'s helpful because it acknowledges that different applications have different needs. ]{#kobo.159.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ A social network and a text editor shouldn\'t necessarily be organized the same way. ]{#kobo.160.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ It\'s frustrating because it doesn\'t give you a starting point. ]{#kobo.161.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ The structure matters more than you might think. ]{#kobo.162.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ A good structure makes it easy to find things. ]{#kobo.163.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ A poor structure means you spend cognitive energy just navigating the codebase. ]{#kobo.164.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When you\'re training an LLM on your codebase or using AI pair programming tools, structure becomes part of the context window. ]{#kobo.165.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Clear structure helps the AI understand your application. ]{#kobo.166.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ How do we render our application? ]{#kobo.167.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_23 .heading-2}

[ The term \" ]{#kobo.168.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} *[ rendering ]{#kobo.169.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}* [ \" itself has multiple meanings in React, which creates confusion. ]{#kobo.170.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When we talk about rendering strategy in React, we mean: where and when does our code execute to determine what the UI should look like? ]{#kobo.171.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ The simplest option is ]{#kobo.172.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} **[ client-side rendering (CSR) ]{#kobo.173.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : all your React code runs in the browser. ]{#kobo.174.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ The server sends a minimal HTML file, JavaScript downloads and executes, and React builds the UI. ]{#kobo.175.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ This is perfect for applications behind authentication or for internal tools. ]{#kobo.176.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ But what if you want to make it as easy as possible for search engines to crawl your application? ]{#kobo.177.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ Then you might need ]{#kobo.177.2 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} **[ server-side rendering (SSR) ]{#kobo.178.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : your React code executes on the server to generate HTML, which is sent to the browser. ]{#kobo.179.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ The browser displays it immediately (good for perceived performance), then downloads the JavaScript and \"hydrates\" the page to make it interactive. ]{#kobo.180.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ Or perhaps your content doesn\'t change often, so you can use ]{#kobo.181.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} **[ static site generation (SSG) ]{#kobo.182.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ , pre-render HTML at build time, and serve it from a CDN. ]{#kobo.183.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ Or perhaps you need all three for different parts of your application. ]{#kobo.184.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ React and its various frameworks support all of these, but you need to decide which to use and where. ]{#kobo.185.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ React Server Components add another dimension to this question. ]{#kobo.186.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Some components can run only on the server, never sending their code to the client. ]{#kobo.187.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ This can sometimes help bundle size but creates new rules about what those components can and can\'t do, and may send more data in the HTML than is saved in the bundle. ]{#kobo.188.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ How do we manage state? ]{#kobo.189.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_24 .heading-2}

[ React ]{#kobo.190.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_4bf7c453 .index-entry index-entry="useState"} [ provides ]{#kobo.191.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} **[ useState ]{#kobo.192.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ and ]{#kobo.193.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} **[ useReducer ]{#kobo.194.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ for ]{#kobo.195.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_1304465d .index-entry index-entry="useReducer"} [ local state, and the Context API for sharing state across components. ]{#kobo.196.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ For many applications, this is enough. ]{#kobo.197.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ For many others, it is not. ]{#kobo.198.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ The ]{#kobo.199.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_1cb3096d .index-entry index-entry="React architecture:challenge"} [ challenge is that \"state\" is not one thing. ]{#kobo.200.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ There\'s local UI state (is this dropdown open?), form state (what has the user typed?), global application state (who is the current user?), server state (what data did we fetch from the API?), and URL state (what page are we on, what are the search parameters?). ]{#kobo.201.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ Each of these has different characteristics and different requirements. ]{#kobo.202.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Local UI state is usually simple. ]{#kobo.203.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Server state is complex because it involves caching, invalidation, background updates, and error handling. ]{#kobo.204.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Treating all state the same leads to problems. ]{#kobo.205.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ The ecosystem offers many solutions: Redux, MobX, Zustand, Jotai, Recoil, XState, and more. ]{#kobo.206.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Then there are specialized tools for server state like TanStack Query (React Query) and SWR. ]{#kobo.207.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Each has different philosophies and trade-offs. ]{#kobo.208.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ How do we style components? ]{#kobo.209.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_25 .heading-2}

[ Some developers prefer traditional CSS files. ]{#kobo.210.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Others prefer CSS Modules for scoping. ]{#kobo.211.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Others prefer utility-first CSS like Tailwind. ]{#kobo.212.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Others prefer CSS-in-JS solutions, and more. ]{#kobo.213.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ The choice affects more than just developer preference. ]{#kobo.214.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ It affects performance (build-time vs. ]{#kobo.215.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ runtime), how you think about component reusability, how easily you can maintain consistent design, and your bundle size. ]{#kobo.216.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ How do we handle data fetching? ]{#kobo.217.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_26 .heading-2}

[ React doesn\'t have built-in data fetching. ]{#kobo.218.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Instead, you have several options: ]{#kobo.219.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- [ You could use the browser\'s fetch API directly ]{#kobo.220.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}
- [ You could use a library like Axios ( ]{#kobo.221.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [[ [ https://axios-http.com/ ]{#kobo.222.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} ]{.url}](https://axios-http.com/){style="text-decoration: none;"} [ ) ]{#kobo.223.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}
- [ You could use a specialized tool like TanStack Query ( ]{#kobo.224.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [[ [ https://tanstack.com/query/latest ]{#kobo.225.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} ]{.url}](https://tanstack.com/query/latest){style="text-decoration: none;"} [ ) that handles caching and background updates ]{#kobo.226.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}
- [ You could use the data fetching capabilities of a meta-framework like Next.js ( ]{#kobo.227.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [[ [ https://nextjs.org/ ]{#kobo.228.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} ]{.url}](https://nextjs.org/){style="text-decoration: none;"} [ ) ]{#kobo.229.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ Where you fetch matters too. ]{#kobo.230.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ In the component? ]{#kobo.231.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ In a custom hook? ]{#kobo.231.2 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ In a separate data layer? ]{#kobo.231.3 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ On the server? ]{#kobo.231.4 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ Each approach has implications for how you test your code and how you handle loading and error states. ]{#kobo.231.5 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ How do we handle authentication? ]{#kobo.232.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_27 .heading-2}

[ Authentication ]{#kobo.233.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_baad5ac6 .index-entry index-entry="React architecture:challenge"} [ intersects with several other decisions. ]{#kobo.234.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Are you using token-based auth (where tokens are stored in ]{#kobo.235.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} ` `{.inlineCode}[` localStorage `{.inlineCode}]{#kobo.236.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}` `{.inlineCode} [ or ]{#kobo.237.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} ` `{.inlineCode}[` sessionStorage `{.inlineCode}]{#kobo.238.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}` `{.inlineCode} [ )? ]{#kobo.239.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ Or cookie-based auth (where ]{#kobo.239.2 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} ` `{.inlineCode}[` httpOnly `{.inlineCode}]{#kobo.240.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}` `{.inlineCode} [ cookies are automatically included with requests)? ]{#kobo.241.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ Cookie-based auth is more secure but requires server-side session handling. ]{#kobo.241.2 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Token-based auth is simpler but vulnerable to XSS attacks if not handled carefully. ]{#kobo.242.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ How do you protect routes? ]{#kobo.243.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ How do you handle authentication errors? ]{#kobo.243.2 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ How do you handle token refresh? ]{#kobo.243.3 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ These patterns need to be consistent across your application. ]{#kobo.244.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ How do we test a React application? ]{#kobo.245.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_28 .heading-2}

[ React applications can be tested at multiple levels: tests for individual functions, tests for components, end-to-end tests for full user flows, and regression tests for UI changes. ]{#kobo.246.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ But time is not an infinite resource. ]{#kobo.247.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Which tests give you the most confidence with the least maintenance burden? ]{#kobo.248.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ Do you test implementation details or user behavior? ]{#kobo.248.2 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ How do you test components that fetch data? ]{#kobo.248.3 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ How do you test authenticated flows? ]{#kobo.248.4 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ The meta-decision ]{#kobo.250.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_29 .heading-2}

# [ Thinking about architectural decisions ]{#kobo.255.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h1_30 .heading-1}

[ Not all architectural ]{#kobo.256.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_425e0f06 .index-entry index-entry="architectural decisions"} [ decisions are equal. ]{#kobo.257.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Some are clearly wrong in almost any context. ]{#kobo.258.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Some are clearly right. ]{#kobo.259.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Most fall somewhere in between, where the right answer depends on your specific situation. ]{#kobo.260.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Let\'s look at some examples. ]{#kobo.261.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ Patterns that usually don\'t work ]{#kobo.262.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_31 .heading-2}

[ Some architectural patterns appear reasonable at first but cause problems as applications grow. ]{#kobo.263.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Here are some seen in React codebases and why they break down: ]{#kobo.264.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Everything in One Place ]{#kobo.265.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : Putting all your components in a single flat folder works fine when you have 10 components. ]{#kobo.266.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ By 50 components, finding anything becomes a chore. ]{#kobo.267.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ By 100 components, it\'s a disaster. ]{#kobo.268.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ The same goes for putting your entire application in a single file. ]{#kobo.269.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ The fact that you can do something doesn\'t mean you should. ]{#kobo.270.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end}

  [ Why doesn\'t this work? ]{#kobo.271.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ Because humans are bad at dealing with long, undifferentiated lists. ]{#kobo.271.2 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ We need structure to make sense of things. ]{#kobo.272.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Your file system is part of your application\'s documentation. ]{#kobo.273.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ It should communicate structure, not hide it. ]{#kobo.274.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ One ]{#kobo.275.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** **[ Big Component ]{#kobo.276.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : A component that does everything seems efficient at first. ]{#kobo.277.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ You have all the logic in one place. ]{#kobo.278.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ But these components become unmaintainable. ]{#kobo.279.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ They\'re hard to test because you have to set up the entire world that surrounds the component. ]{#kobo.280.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ They\'re hard to reuse because they\'re coupled to too many things. ]{#kobo.281.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ They\'re hard to reason about because you have to hold too much in your head at once. ]{#kobo.282.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Large components ]{#kobo.283.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : When you\'re using AI to modify code, large components create problems. ]{#kobo.284.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ The component might fill much of the AI\'s context window, or the AI might suggest changes that don\'t account for subtle interactions in the component. ]{#kobo.285.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Everything global ]{#kobo.286.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : Putting all your state in a global store seems like it would make things simple. ]{#kobo.287.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Everything in one place! ]{#kobo.288.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ But this actually makes things harder. ]{#kobo.288.2 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When any component can modify any state, understanding data flow becomes impossible. ]{#kobo.289.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ You can\'t look at a component and understand what state it depends on. ]{#kobo.290.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Changes become risky because you don\'t know what else might break. ]{#kobo.291.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Global state has its place, but most state should be local. ]{#kobo.292.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Keep state as close as possible to where it\'s used, and lift it only when you need to share it. ]{#kobo.293.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Wrong Tool, Right Problem ]{#kobo.294.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : The React ecosystem has specialized tools for specialized problems. ]{#kobo.295.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Using a generic global state manager to handle server data (data fetched from an API) means reimplementing caching, revalidation, optimistic updates, and background sync. ]{#kobo.296.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ These are complex problems. ]{#kobo.297.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Tools like TanStack Query solve them, and they\'re proven at scale. ]{#kobo.298.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Ignoring specialized tools to avoid adding dependencies sounds pragmatic, but you end up with more code, more bugs, and more maintenance burden. ]{#kobo.299.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ A carefully chosen dependency is the solution, not the problem. ]{#kobo.300.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Ignoring Security ]{#kobo.301.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : Not ]{#kobo.302.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_99ff47bd .index-entry index-entry="architectural decisions"} [ sanitizing user input is not an architectural decision, it\'s a mistake. ]{#kobo.303.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ If your application displays user-generated content (and we\'ll be building one that does), you must sanitize it. ]{#kobo.304.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Otherwise, you\'re vulnerable to attacks where malicious users inject scripts that execute in other users\' browsers. ]{#kobo.305.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ This is especially important if you\'re using Markdown or rich text. ]{#kobo.306.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Ignoring Performance Infrastructure ]{#kobo.307.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : Serving your application from a single server in one geographic location means users far from that server experience slow load times. ]{#kobo.308.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ This is a solved problem: content delivery networks (CDNs) distribute your application globally. ]{#kobo.309.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Users are served from the edge location closest to them. ]{#kobo.310.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ This is table stakes for modern web applications. ]{#kobo.311.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

## [ Patterns that usually work ]{#kobo.312.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h2_32 .heading-2}

[ On the other hand, certain patterns have proven themselves across teams and codebases of all sizes. ]{#kobo.313.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ These aren\'t rules; they\'re decisions that tend to serve you well unless you have a good reason to deviate: ]{#kobo.314.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Feature-Based Structure ]{#kobo.315.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : Organizing code by feature rather than by technical role scales better. ]{#kobo.316.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Instead of a folder for all components, a folder for all hooks, and a folder for all API calls, you have a folder for user authentication, a folder for the ideas feature, a folder for reviews. ]{#kobo.317.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Each folder contains everything related to that feature, and teams can work within separate features with fewer merge conflicts. ]{#kobo.318.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Why does this work? ]{#kobo.319.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ Because features are how we think about applications. ]{#kobo.319.2 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When you need to modify how ideas work, you go to the ideas folder. ]{#kobo.320.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Everything you need is co-located. ]{#kobo.321.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When a new developer joins, they can understand the application by understanding its features. ]{#kobo.322.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ This also helps with AI-assisted development. ]{#kobo.323.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When an LLM is helping you modify a feature, all the relevant context is in one place, making it easier for the AI to understand the full scope of what it\'s working with. ]{#kobo.324.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Co-located State ]{#kobo.325.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : Keeping state as local as possible. ]{#kobo.326.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ If only one component needs it, use useState in that component. ]{#kobo.327.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ If several nearby components need it, lift it to their common parent. ]{#kobo.328.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Only put state in a global store if it truly needs to be global (user authentication, theme preferences, etc.). ]{#kobo.329.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ This keeps the scope of state clear. ]{#kobo.330.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When you look at a component, you can see what state it depends on. ]{#kobo.331.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Changes are less risky because the impact is localized. ]{#kobo.332.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Appropriate Specialization ]{#kobo.333.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : Use specialized tools for specialized problems. ]{#kobo.334.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Use TanStack Query for server state. ]{#kobo.335.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Use React Hook Form for complex forms. ]{#kobo.336.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Use Zod for validation. ]{#kobo.337.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ These tools exist because they solve hard problems well. ]{#kobo.338.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Your application-specific code should focus on application-specific problems, not on reimplementing generic solutions. ]{#kobo.339.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Layered Defense ]{#kobo.340.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : Security is not a single decision; it\'s a series of decisions. ]{#kobo.341.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Use httpOnly cookies for authentication tokens so they can\'t be accessed by JavaScript. ]{#kobo.342.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Validate input on both client and server. ]{#kobo.343.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Sanitize user-generated content before rendering. ]{#kobo.344.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Use TypeScript to catch type errors at compile time. ]{#kobo.345.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Use ESLint to catch common mistakes. ]{#kobo.346.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Each layer catches things the others miss. ]{#kobo.347.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Small, Focused Components ]{#kobo.348.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : Components should do one thing. ]{#kobo.349.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ If a component is getting ]{#kobo.350.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_66e372d4 .index-entry index-entry="architectural decisions"} [ large, that\'s a signal to break it apart. ]{#kobo.351.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Small components are easier to understand, test, modify, and reuse. ]{#kobo.352.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ They are also easier for AI to work with, since they fit more easily in context windows and their behavior is clearer. ]{#kobo.353.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Tooling and Analysis ]{#kobo.354.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ : Use tools to enforce consistency, like ESLint for code quality, Prettier for formatting, and TypeScript for type safety. ]{#kobo.355.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ These tools catch mistakes before code runs. ]{#kobo.356.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ They reduce cognitive load because you don\'t have to remember rules. ]{#kobo.357.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ They create consistency across the team because the tools enforce it. ]{#kobo.358.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ This is particularly valuable when using AI code generation tools. ]{#kobo.359.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ When the AI suggestions are automatically formatted and type-checked, you catch issues immediately. ]{#kobo.360.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end}

  [ These patterns and anti-patterns give us a menu of architectural possibilities. ]{#kobo.361.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ But patterns in the abstract only get you so far. ]{#kobo.362.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ To see how these ideas play out in practice, we need a real application with real requirements. ]{#kobo.363.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ That\'s what we\'ll plan next. ]{#kobo.364.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

# [ Planning our application ]{#kobo.365.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h1_33 .heading-1}

### [ Architectural decisions ]{#kobo.550.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h3_37 .heading-3}

[ Now we ]{#kobo.551.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_885562fa .index-entry index-entry="data model:architectural decisions"} [ can make informed decisions. ]{#kobo.552.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ We\'ve defined what the application does (functional requirements), how well it needs to do it (non-functional requirements), and what data it works with (the data model). ]{#kobo.553.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Each architectural decision below is a direct response to these requirements. ]{#kobo.554.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

### [ Project structure: feature-based ]{#kobo.557.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h3_38 .heading-3}

[ We\'ll ]{#kobo.558.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_2a3a6ddc .index-entry index-entry="data model:feature-based project structure"} [ organize by domain features (users, ideas, reviews) rather than by technical role. ]{#kobo.559.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Each feature folder contains its components, API calls, hooks, types, and utilities. ]{#kobo.560.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Shared code (auth, UI components, utilities) lives in shared folders. ]{#kobo.561.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

### [ Rendering strategy: hybrid ]{#kobo.566.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h3_39 .heading-3}

[ Different pages ]{#kobo.567.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_e8b98173 .index-entry index-entry="data model:hybrid rendering strategy"} [ have different needs. ]{#kobo.568.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ We\'ll use React Router v7, which supports the following: ]{#kobo.569.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- **[ Static Site Generation (SSG) ]{#kobo.570.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ for the landing page: pre-rendered at build time, instant loading, perfect for marketing ]{#kobo.571.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}
- **[ Server-Side Rendering (SSR) ]{#kobo.572.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ for public idea pages: rendered on-demand on the server, good for SEO and initial load performance ]{#kobo.573.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}
- **[ Client-Side Rendering (CSR) ]{#kobo.574.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}** [ for authenticated pages: rich interactivity, fast navigation after initial load ]{#kobo.575.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ We\'ll also use code splitting to lazy load routes, prefetching for critical resources, TanStack Query for caching API responses, and skeleton screens for loading states. ]{#kobo.576.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

[ React Router v7 makes this straightforward. ]{#kobo.577.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ We can use a hybrid rendering approach, which means each route can specify its own rendering method. ]{#kobo.578.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

### [ State management: Multi-tier ]{#kobo.579.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h3_40 .heading-3}

[ Different ]{#kobo.580.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_f165325a .index-entry index-entry="data model:multi-tier state management"} [ types of state need different tools: ]{#kobo.581.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- [ Local state (component-specific): useState and useReducer ]{#kobo.582.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

- [ Server state (API data): TanStack Query ]{#kobo.584.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}
- [ Form state: React Hook Form with Zod validation ]{#kobo.585.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

### [ Styling: Tailwind CSS + BaseUI ]{#kobo.590.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h3_41 .heading-3}

[ We\'ll use ]{#kobo.591.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_f5cf735e .index-entry index-entry="data model:Tailwind CSS + BaseUI styling"} [ Tailwind CSS for utility-first styling. ]{#kobo.592.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ It\'s fast (build-time, not runtime), it enforces consistency through design tokens, and its purge feature keeps bundles small. ]{#kobo.593.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ Developer experience is excellent once you learn the utilities. ]{#kobo.594.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}

### [ Authentication: Cookie-based ]{#kobo.600.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} {#h3_42 .heading-3}

[ We\'ll ]{#kobo.601.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{#idx_f1c4d706 .index-entry index-entry="data model:cookie-based authentication"} [ use httpOnly cookies for authentication. ]{#kobo.602.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ The cookie is set by the server and automatically included with requests. ]{#kobo.603.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"} [ ]{.sentence-end} [ JavaScript can\'t access it, which protects against XSS attacks. ]{#kobo.604.1 .koboSpan xmlns="http://www.w3.org/1999/xhtml"}
