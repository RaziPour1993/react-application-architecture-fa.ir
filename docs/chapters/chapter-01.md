# فصل ۱: درک معماری اپلیکیشن‌های React

React محبوب‌ترین کتابخانه JavaScript در جهان برای ساخت رابط کاربری است. بازار کار پر از فرصت‌های مبتنی بر React است.

React به دلایل متعددی این‌قدر محبوب شده: API کوچک، اکوسystem عظیم کامپوننت‌های آماده، و انعطاف‌پذیری.

با این حال برای توسعه‌دهندهٔ امروزی، همین انعطاف می‌تواند شمشیر دو لبه باشد. React نظر قطعی دربارهٔ نحوهٔ ساختاردهی اپلیکیشن نمی‌دهد، پس هر تیم باید خودش راهش را پیدا کند.

این منجر به ناسازگاری بین پروژه‌ها، تصمیم‌های معماری تحت فشار deadline بدون درک trade-offها، و codebaseهایی می‌شود که با رشد، نگهداری‌شان سخت‌تر می‌شود.

این یعنی هر تیم با سؤالات مهمی روبه‌روست: اپلیکیشن را چگونه ساختاربندی کنید؟ state چگونه جریان می‌یابد؟ routing، استایل کامپوننت‌ها و هر تصمیم معماری دیگر را چطور بگیرید؟ React پاسخ قطعی نمی‌دهد.

با این حال، تصمیم‌های معماری خوب تفاوت بین ساخت یک اپ قابل نگهداری و ایجاد spaghetti code سخت‌دیباگ است.

هدف این کتاب همان شفافیتی است که جا افتاده: چگونه اپلیکیشن React را طوری معماری کنید که مقیاس‌پذیر، قابل نگهداری و همراه با رشد شما و تیم‌تان باشد.

در این فصل موارد زیر را پوشش می‌دهیم:

- بحث دربارهٔ مزایای داشتن معماری خوب اپلیکیشن
- بررسی چالش‌های معماری اپلیکیشن‌های React
- درک تصمیم‌های معماری هنگام ساخت اپلیکیشن React
- برنامه‌ریزی اپلیکیشنی که در این کتاب می‌سازیم

در پایان این فصل، می‌دانید چگونه از منظر معماری به اپلیکیشن‌های React نگاه کنید و تصمیم‌های آگاهانه‌ای بگیرید که موفقیت بلندمدت پروژه را تضمین کند.

## چرا معماری مهم است {#h1_15}

هر اپلیکیشنی معماری دارد — حتی آن‌هایی که کسی به آن فکر نکرده. ممکن است پروژهٔ Reactای باز کنید و ۴۷ کامپوننت در یک پوشهٔ `components/`، فراخوانی‌های API پراکنده در فایل‌های تصادفی، و state که بسته به نویسندهٔ کد سه روش مختلف مدیریت شده ببینید.

این هم معماری است. فقط عمدی نبوده.

سؤال این نیست که اپلیکیشن شما معماری دارد یا نه؛ سؤال این است که آن معماری عمدی انتخاب شده یا تصادفی شکل گرفته.

وقتی از معماری حرف می‌زنیم، دربارهٔ تصمیم‌های سطح بالا هستیم که شکل کار اپلیکیشن را تعیین می‌کنند. کد چگونه سازمان‌دهی شده؟ بخش‌های مختلف چگونه با هم ارتباط دارند؟ تیم چه الگوهایی را دنبال می‌کند؟ این تصمیم‌ها بعداً سخت عوض می‌شوند چون همهٔ چیزهای ساخته‌شده روی آن‌ها استوارند.

در این فصل می‌بینیم چرا معماری اصلاً مهم است، چالش‌های خاص React چیست، الگوهای مؤثر و ناکارآمد کدام‌اند، و اپ واقعی‌ای که در ادامهٔ کتاب می‌سازیم را برنامه‌ریزی می‌کنیم.

### معماری به‌عنوان بنیان {#h2_16}

وقتی ساختمانی می‌سازید، از پی شروع می‌کنید. دیوار را اول نمی‌سازید و بعد پی را فکر نمی‌کنید. نرم‌افزار هم همین‌طور است، اما پیامدها فوری نیست.

می‌توانید بدون فکر به معماری feature بسازید و مدتی همه‌چیز خوب پیش برود. مشکل وقتی پدیدار می‌شود که بخواهید چیزی را عوض کنید.

بنیان معماری محکم، اپلیکیشن را در برابر تغییر مقاوم می‌کند. و تغییر تنها ثابت نرم‌افزار است.

نیازمندی‌ها عوض می‌شوند. feature جدید اضافه می‌شود. اعضای تیم می‌آیند و می‌روند. فناوری‌های جدید ظاهر می‌شوند. اولویت‌های کسب‌وکار می‌چرخد.

اپلیکیشنی که روی بنیان محکم ساخته شده می‌تواند با این تغییرات adapt شود. اپی روی بنیان لرزان به workaroundهای گران‌تر و در نهایت به گران‌ترین گزینه ختم می‌شود: بازنویسی کامل.

### هماهنگی و سرعت توسعه {#h2_17}

معماری خوب کار تیمی را آسان‌تر می‌کند. وقتی کامپوننت‌ها مرز و مسئولیت روشن دارند، توسعه‌دهندگان روی بخش‌های مختلف بدون تداخل کار می‌کنند. کار را بهتر بین اعضای تیم تقسیم می‌کنید. تخمین دقیق‌تری می‌دهید چون scope تغییرات را می‌فهمید.

با بزرگ شدن تیم اهمیتش بیشتر می‌شود. یک توسعه‌دهندهٔ solo می‌تواند کل اپ بد معماری را در ذهنش نگه دارد، اما این scale نمی‌شود. وقتی نفر دوم می‌پیوندد باید الگوها را بفهمد. وقتی نفر سوم می‌پیوندد باید الگوها یکسان باشند. با پنج نفر، معماری ناهمگون اصطکاک دائمی ایجاد می‌کند.

معماری بد روی توسعه با AI هم اثر می‌گذارد. وقتی از ابزارهایی مثل GitHub Copilot یا Claude استفاده می‌کنید، آن‌ها بر اساس دادهٔ آموزشی و الگوهای codebase شما پیشنهاد می‌دهند. معماری یکدست یعنی پیشنهادهای یکدست. معماری ناهمگون یعنی AI نمی‌داند کدام الگو را دنبال کند و زمان بیشتری صرف اصلاح پیشنهادها می‌شود تا آنچه از استفادهٔ آن صرفه‌جویی کردید.

### سرعت از طریق شفافیت {#h2_18}

برخلاف برداشت اولیه، وقت گذاشتن روی معماری از ابتدا سرعت توسعه را بالا می‌برد. وقتی تصمیم‌های معماری گرفته شده، توسعه‌دهندگان روی حل مسائل کسب‌وکار تمرکز می‌کنند نه بحث فنی برای هر feature. feature جدید در الگوهای موجود جا می‌گیرد. تازه‌واردها سریع‌تر productive می‌شوند چون ساختار روشن است.

جایگزین، چیزی است که می‌توانیم *architecture decision fatigue* بنامیم. هر feature جدید نیاز به تصمیم دارد: چگونه کار کند، کجا باشد، چگونه با بخش‌های دیگر ارتباط بگیرد. این micro-decisionها جمع می‌شوند. ناسازگاری ایجاد می‌کنند. همه‌چیز را کند می‌کنند.

این موضوع هنگام کار با LLM برای code generation به‌خصوص مهم است. اگر از AI بخواهید feature به اپ well-architected اضافه کند، می‌تواند الگوهای موجود را دنبال کند. اگر codebase ناهمگون باشد باید حدس بزند کدام الگو را دنبال کند — و اغلب اشتباه حدس می‌زند.

### اقتصاد معماری خوب {#h2_19}

پروژه‌های نرم‌افزاری هزینه دارند. بیشتر این هزینه مربوط به افراد است: زمان، تلاش، توجه.

معماری خوب با مؤثرتر کردن افراد هزینه را کم می‌کند. زمان debug کمتر می‌شود. زمان جستجو برای یافتن چیزها کمتر می‌شود. زمان دور زدن محدودیت‌های معماری کمتر می‌شود.

معماری ضعیف هزینهٔ compounding دارد. هر feature جدید طولانی‌تر می‌شود چون باید دور مشکلات موجود کار کنید. هر باگ سخت‌تر fix می‌شود چون مرز بین componentها نامشخص است. در نهایت افزودن feature غیرممکن و بازنویسی لازم می‌شود. بازنویسی‌ها گران و پرریسک‌اند.

معماری خوب در تصمیم‌های کسب‌وکار هم کمک می‌کند. وقتی ساختار اپلیکیشن را می‌فهمید، هزینه را دقیق‌تر تخمین می‌زنید. می‌توانید تصمیم بگیرید کدام feature را بسازید و کدام را به بعد موکول کنید.

### معماری و کیفیت محصول {#h2_20}

وقتی توسعه‌دهندگان کمتر با معماری می‌جنگند، وقت بیشتری برای تمرکز روی نیاز کاربر دارند. می‌توانند روی polish وقت بگذارند. دربارهٔ edge case فکر کنند. error handling را بهبود دهند. اپلیکیشن را accessibleتر کنند.

معماری ضعیف چرخهٔ vicious ایجاد می‌کند. توسعه‌دهندگان وقتشان صرف fix مشکلات معماری می‌شود. featureها با لبه‌های خشن ship می‌شوند. کاربران باگ می‌بینند. توسعه‌دهندگان وقت بیشتری صرف fix اضطراری می‌کنند. هیچ وقت برای بهبود معماری نیست چون همیشه چیز فوری‌ای هست.

معماری خوب چرخهٔ مثبت بهبود کیفیت ایجاد می‌کند. توسعه‌دهندگان وقتشان صرف ساخت feature می‌شود. featureها با کیفیت بهتر ship می‌شوند. کاربران راضی‌ترند. باگ کمتری برمی‌گردد. تیم وقت دارد معماری را سالم نگه دارد.

هدف هر نرم‌افزار حل مسئلهٔ واقعی برای انسان واقعی است. همهٔ چیز دیگر در خدمت همان هدف است. معماری مهم است چون تعیین می‌کند روی همان goal بمانید یا در technical debt غرق شوید.

## چالش معماری در React {#h1_21}

React ابزار عالی برای ساخت رابط کاربری است، اما عمداً بعضی مسائل را حل نمی‌کند. فهمیدن اینکه React چه می‌کند و چه نمی‌کند، به درک تصمیم‌های معماری لازم کمک می‌کند.

انعطاف React هم برکت است هم نفرین. برکت است چون کتابخانه شما را مجبور نمی‌کند الگویی را بپذیرید که با مسئله‌تان نمی‌خواند. نفرین است چون مسئولیت هر تصمیم با شماست. این به محبوبیت React کمک کرده و اکوسystem غنی کتابخانه‌ها و ابزارها ساخته — که می‌توانید آن‌ها را تصمیم‌های از پیش گرفته‌شده بدانید. اما مشکل جدیدی هم ایجاد کرده: *choice overload*.

این‌ها سؤالاتی‌اند که هر تیم توسعه با شروع پروژهٔ React جدید با آن‌ها روبه‌روست:

### چگونه پروژه را ساختاربندی کنیم؟ {#h2_22}

React نظری دربارهٔ file structure ندارد. می‌توانید:

- همه‌چیز را در یک فایل بگذارید (هرچند نباید!)
- پوشهٔ flat از componentها داشته باشید
- بر اساس نقش فنی سازمان‌دهی کنید (همهٔ componentها در یک پوشه، همهٔ hookها در پوشهٔ دیگر)
- بر اساس feature سازمان‌دهی کنید
- بر اساس مفهوم domain سازمان‌دهی کنید

Dan Abramov، یکی از اعضای سابق core team React، معروف است گفته: *«Move files around until it feels right.»* برای بعضی راهنماست، برای بعضی frustrating. راهنماست چون می‌پذیرد اپلیکیشن‌های مختلف نیاز متفاوت دارند. شبکهٔ اجتماعی و text editor لزوماً نباید یک‌سان سازمان‌دهی شوند. frustrating است چون نقطهٔ شروع نمی‌دهد.

ساختار مهم‌تر از آن است که فکر کنید. ساختار خوب یافتن را آسان می‌کند. ساختار بد یعنی انرژی cognitive صرف navigation در codebase می‌شود. وقتی LLM را روی codebase آموزش می‌دهید یا از AI pair programming استفاده می‌کنید، ساختار بخشی از context window می‌شود. ساختار روشن به AI کمک می‌کند اپلیکیشن را بفهمد.

### چگونه اپلیکیشن را رندر کنیم؟ {#h2_23}

اصطلاح *rendering* در React چند معنا دارد که گیج‌کننده است. وقتی از استراتژی rendering در React حرف می‌زنیم، منظور این است: کد کجا و کی اجرا می‌شود تا UI مشخص شود؟

ساده‌ترین گزینه **client-side rendering (CSR)** است: همهٔ کد React در مرورگر اجرا می‌شود. سرور HTML مینیمال می‌فرستد، JavaScript دانلود و اجرا می‌شود، React UI را می‌سازد. برای اپلیکیشن‌های پشت authentication یا ابزارهای داخلی عالی است.

اما اگر بخواهید crawl برای موتورهای جستجو تا حد ممکن آسان باشد؟ آن وقت شاید به **server-side rendering (SSR)** نیاز داشته باشید: کد React روی server اجرا می‌شود و HTML تولید می‌کند که به مرورگر فرستاده می‌شود. مرورگر فوراً نشان می‌دهد (برای perceived performance خوب است)، سپس JavaScript را دانلود می‌کند و صفحه را hydrate می‌کند تا interactive شود.

یا شاید محتوا زیاد عوض نمی‌شود و بتوانید از **static site generation (SSG)** استفاده کنید: HTML در build time pre-render می‌شود و از CDN سرو می‌شود.

یا شاید هر سه را برای بخش‌های مختلف اپلیکیشن نیاز داشته باشید. React و frameworkهای مختلف از همه پشتیبانی می‌کنند، اما باید تصمیم بگیرید کجا کدام را استفاده کنید.

React Server Components بعد دیگری به این سؤال اضافه می‌کند. بعضی componentها فقط روی server اجرا می‌شوند و کدشان هرگز به client فرستاده نمی‌شود. گاهی به bundle size کمک می‌کند، اما rule جدیدی دربارهٔ آنچه آن componentها می‌توانند و نمی‌توانند انجام دهند ایجاد می‌کند و ممکن است دادهٔ بیشتری در HTML بفرستد نسبت به آنچه در bundle صرفه‌جویی می‌شود.

### چگونه state را مدیریت کنیم؟ {#h2_24}

React **`useState`** و **`useReducer`** برای state محلی، و Context API برای اشتراک state بین componentها فراهم می‌کند. برای بسیاری از اپلیکیشن‌ها کافی است. برای بسیاری دیگر نه.

چالش این است که *state* یک چیز واحد نیست. state محلی UI (dropdown باز است؟)، state فرم (کاربر چه تایپ کرده؟)، state سراسری اپلیکیشن (کاربر فعلی کیست؟)، server state (چه داده‌ای از API گرفتیم؟)، و URL state (روی چه صفحه‌ایم، search parameterها چیست؟) وجود دارد.

هر کدام ویژگی و نیاز متفاوت دارند. state محلی UI معمولاً ساده است. server state پیچیده است چون cache، invalidation، background update و error handling دارد. یکسان دیدن همهٔ stateها به مشکل ختم می‌شود.

اکوسystem راه‌حل‌های زیادی دارد: Redux، MobX، Zustand، Jotai، Recoil، XState و بیشتر. سپس ابزارهای تخصصی server state مثل TanStack Query (React Query) و SWR. هر کدام فلسفه و trade-off متفاوت دارند.

### چگونه کامپوننت‌ها را استایل دهیم؟ {#h2_25}

بعضی توسعه‌دهندگان CSS سنتی را ترجیح می‌دهند. بعضی CSS Modules برای scoping. بعضی utility-first CSS مثل Tailwind. بعضی CSS-in-JS و بیشتر.

انتخاب فقط سلیقهٔ developer نیست. روی performance (build-time در برابر runtime)، نحوهٔ فکر دربارهٔ reusability component، consistency طراحی، و bundle size اثر دارد.

### چگونه data fetching را انجام دهیم؟ {#h2_26}

React data fetching داخلی ندارد. گزینه‌ها:

- استفاده مستقیم از `fetch` API مرورگر
- استفاده از کتابخانه‌ای مثل [Axios](https://axios-http.com/)
- استفاده از ابزار تخصصی مثل [TanStack Query](https://tanstack.com/query/latest) که cache و background update را handle می‌کند
- استفاده از قابلیت‌های data fetching یک meta-framework مثل [Next.js](https://nextjs.org/)

*کجا* fetch کنید هم مهم است. در component؟ در custom hook؟ در لایهٔ data جدا؟ روی server؟ هر رویکرد روی test و loading/error state اثر دارد.

### چگونه احراز هویت را پیاده‌سازی کنیم؟ {#h2_27}

Authentication با چند تصمیم دیگر تلاقی دارد. از token-based auth استفاده می‌کنید (token در `localStorage` یا `sessionStorage`)؟ یا cookie-based auth (cookieهای `httpOnly` خودکار با request فرستاده می‌شوند)؟ cookie-based امن‌تر است اما session handling سمت server لازم دارد. token-based ساده‌تر است اما در برابر XSS آسیب‌پذیرتر است اگر درست handle نشود.

routeها چگونه محافظت می‌شوند؟ خطاهای authentication چگونه handle می‌شوند؟ token refresh چگونه انجام می‌شود؟

این الگوها باید در کل اپلیکیشن یکسان باشند.

### چگونه اپلیکیشن React را تست کنیم؟ {#h2_28}

اپلیکیشن React را می‌توان در چند سطح تست کرد: تست توابع individual، تست componentها، تست end-to-end برای flow کامل کاربر، و regression test برای تغییرات UI.

اما زمان بی‌نهایت نیست. کدام تست بیشترین اطمینان را با کمترین maintenance burden می‌دهد؟ implementation detail تست می‌کنید یا رفتار کاربر؟ componentهایی که data fetch می‌کنند چطور؟ flowهای authenticated چطور؟

استراتژی تست روی معماری اثر دارد، چون بعضی الگوهای معماری test-friendlyتر از بقیه‌اند.

### تصمیم متا {#h2_29}

زیر همهٔ این‌ها یک تصمیم متا هست: از meta-framework مثل Next.js، React Router یا TanStack Start استفاده می‌کنید؟

این frameworkها بعضی تصمیم‌های بالا را برایتان می‌گیرند، اما به الگوهایشان lock می‌کنید.

لزوماً بد نیست. constraint گاهی آزاد می‌کند تا کمتر معماری فکر کنید و بیشتر روی خود اپ تمرکز کنید.

اما انتخابی است که همهٔ چیز دیگر را تحت تأثیر قرار می‌دهد.

## اندیشیدن درباره تصمیم‌های معماری {#h1_30}

همهٔ تصمیم‌های معماری برابر نیستند. بعضی تقریباً در هر context clearly wrong هستند. بعضی clearly right. بیشتر جایی بین این دو قرار دارند که پاسخ درست به situation خاص شما بستگی دارد. بیایید چند مثال ببینیم.

### الگوهایی که معمولاً جواب نمی‌دهند {#h2_31}

بعضی الگوهای معماری در نگاه اول reasonable به نظر می‌رسند اما با رشد اپلیکیشن مشکل‌ساز می‌شوند. چند نمونه از codebaseهای React و دلیل شکستشان:

- **Everything in One Place**: گذاشتن همهٔ componentها در یک پوشهٔ flat وقتی ۱۰ component دارید خوب کار می‌کند. با ۵۰ component یافتن چیزها chore می‌شود. با ۱۰۰ component فاجعه است. همین‌طور گذاشتن کل اپلیکیشن در یک فایل. اینکه *می‌توانید* کاری را انجام دهید به معنای این نیست که *باید*.

  چرا جواب نمی‌دهد؟ چون انسان‌ها با لیست‌های بلند و بدون تمایز بد کار می‌کنند. برای فهمیدن چیزها به ساختار نیاز داریم. filesystem بخشی از مستندات اپلیکیشن است. باید ساختار را communicate کند نه پنهان کند.

- **One Big Component**: componentی که همه‌کاره است در ابتدا efficient به نظر می‌رسد. همهٔ logic در یک جا است. اما unmaintainable می‌شود. test سخت است چون باید کل world اطراف component را setup کنید. reuse سخت است چون به چیزهای زیادی coupled است. reasoning سخت است چون باید زیاد در ذهن نگه دارید.

- **Large components**: وقتی از AI برای تغییر code استفاده می‌کنید، componentهای بزرگ مشکل‌سازند. component ممکن است بخش زیادی از context window AI را پر کند، یا AI تغییراتی پیشنهاد دهد که تعاملات subtle داخل component را در نظر نگیرد.

- **Everything global**: گذاشتن همهٔ state در store سراسری ساده به نظر می‌رسد. همه‌چیز در یک جا! اما سخت‌تر می‌کند. وقتی هر component می‌تواند هر state را تغییر دهد، فهم data flow غیرممکن می‌شود. نمی‌توانید به component نگاه کنید و بفهمید به چه stateای وابسته است. تغییر پرریسک می‌شود چون نمی‌دانید چه چیز دیگری ممکن است بشکند. state global جای خود را دارد، اما بیشتر state باید local باشد. state را تا جایی که ممکن است نزدیک محل استفاده نگه دارید و فقط وقتی لازم است share کنید.

- **Wrong Tool, Right Problem**: اکوسystem React ابزار تخصصی برای مسائل تخصصی دارد. استفاده از state manager عمومی برای server data (دادهٔ fetch شده از API) یعنی reimplement کردن cache، revalidation، optimistic update و background sync. این‌ها مسائل پیچیده‌اند. ابزارهایی مثل TanStack Query آن‌ها را حل کرده‌اند و در scale اثبات شده‌اند. نادیده گرفتن ابزار تخصصی برای اجتناب از dependency pragmatic به نظر می‌رسد، اما به code بیشتر، bug بیشتر و maintenance بیشتر ختم می‌شود. dependency انتخاب‌شدهٔ دقیق راه‌حل است نه مشکل.

- **Ignoring Security**: sanitize نکردن ورودی کاربر تصمیم معماری نیست — اشتباه است. اگر اپلیکیشن محتوای user-generated نشان می‌دهد (و ما یکی می‌سازیم) باید sanitize کنید. وگرنه در برابر حملاتی آسیب‌پذیرید که کاربران مخرب script inject می‌کنند تا در مرورگر کاربران دیگر اجرا شود. با Markdown یا rich text به‌خصوص مهم است.

- **Ignoring Performance Infrastructure**: سرو اپلیکیشن از یک server در یک موقعیت جغرافیایی یعنی کاربران دور از آن server load time کند تجربه می‌کنند. این مسئله حل شده: CDNها اپلیکیشن را globally توزیع می‌کنند. کاربران از نزدیک‌ترین edge location سرو می‌شوند. برای اپلیکیشن‌های web مدرن table stakes است.

### الگوهایی که معمولاً جواب می‌دهند {#h2_32}

از طرف دیگر، الگوهایی هستند که در تیم‌ها و codebaseهای هر اندازه خود را ثابت کرده‌اند. rule نیستند؛ تصمیم‌هایی‌اند که مگر دلیل خوب برای انحراف داشته باشید، معمولاً به دردتان می‌خورند:

- **Feature-Based Structure**: سازمان‌دهی code بر اساس feature نه نقش فنی بهتر scale می‌شود. به‌جای پوشه برای همهٔ componentها، پوشه برای همهٔ hookها و پوشه برای همهٔ API callها، پوشه برای user authentication، پوشه برای ideas و پوشه برای reviews دارید. هر پوشه همهٔ چیز مربوط به آن feature را دارد و تیم‌ها با merge conflict کمتر روی featureهای جدا کار می‌کنند. چرا جواب می‌دهد؟ چون featureها همان‌طور است که به اپلیکیشن فکر می‌کنیم. وقتی باید نحوهٔ کار ideas را عوض کنید به پوشهٔ ideas می‌روید. همهٔ چیز لازم co-located است. تازه‌وارد با فهم featureها اپ را می‌فهمد. برای AI-assisted development هم کمک می‌کند: وقتی LLM feature را تغییر می‌دهد، context مرتبط در یک جا است.

- **Co-located State**: state تا جایی که ممکن local نگه داشتن. اگر فقط یک component نیاز دارد `useState` در همان component. اگر چند component نزدیک نیاز دارند به parent مشترک lift کنید. فقط اگر واقعاً global لازم است (authentication کاربر، theme preference و...) در store سراسری بگذارید. scope state روشن می‌ماند. به component نگاه می‌کنید و می‌بینید به چه stateای وابسته است. تغییر کمتر risky است چون impact localized است.

- **Appropriate Specialization**: ابزار تخصصی برای مسائل تخصصی. TanStack Query برای server state. React Hook Form برای فرم پیچیده. Zod برای validation. این ابزارها وجود دارند چون مسائل سخت را خوب حل می‌کنند. code مخصوص اپلیکیشن باید روی مسائل مخصوص اپ تمرکز کند نه reimplement راه‌حل‌های generic.

- **Layered Defense**: امنیت یک تصمیم نیست — series of decisions است. `httpOnly` cookie برای token authentication تا JavaScript دسترسی نداشته باشد. validation در client و server. sanitize محتوای user-generated قبل از render. TypeScript برای catch خطای type در compile time. ESLint برای catch اشتباهات رایج. هر لایه چیزهایی را می‌گیرد که بقیه از دست می‌دهند.

- **Small, Focused Components**: componentها باید یک کار انجام دهند. اگر component بزرگ می‌شود سیگنال شکستن است. componentهای کوچک understand، test، modify و reuse آسان‌ترند. برای AI هم راحت‌ترند چون در context window جا می‌شوند و رفتارشان روشن‌تر است.

- **Tooling and Analysis**: ابزار برای enforce consistency — ESLint برای code quality، Prettier برای formatting، TypeScript برای type safety. این ابزارها خطا را قبل از run catch می‌کنند. cognitive load کم می‌کنند چون ruleها را به خاطر نمی‌سپارید. consistency بین تیم ایجاد می‌کنند. برای AI code generation به‌خصوص valuable است: وقتی پیشنهاد AI خودکار format و type-check می‌شود، issue فوراً دیده می‌شود.

  این الگوها و ضدالگوها منویی از امکانات معماری به ما می‌دهند. اما الگو به‌تنهایی فقط تا اینجا پیش می‌برد. برای دیدن عملی این ایده‌ها به اپلیکیشن واقعی با نیازمندی واقعی نیاز داریم. همان را برنامه‌ریزی می‌کنیم.

## برنامه‌ریزی اپلیکیشن {#h1_33}

بحث انتزاعی معماری مفید است، اما تصمیم‌های معماری در بستر اپلیکیشن واقعی با نیازمندی واقعی گرفته می‌شوند. پس یکی می‌سازیم.

در این کتاب **AI Ideas Community Platform** می‌سازیم — اپلیکیشنی اجتماعی که کاربران ایده‌های اپلیکیشن AI را به اشتراک می‌گذارند، ایده‌های دیگران را مرور می‌کنند و برایشان نقد می‌نویسند. این یک مرکز جامعه برای کسانی است که می‌خواهند آنچه در فضای AI می‌سازند — یا فکر می‌کنند بسازند — را به اشتراک بگذارند.

چرا این اپلیکیشن؟ چون بیشتر تصمیم‌های معماری که بحث کردیم را لمس می‌کند. صفحات عمومی دارد (برای SEO). صفحات احراز هویت‌شده دارد (برای ارسال ایده و نقد). محتوای تولیدشده توسط کاربر دارد (یعنی sanitization لازم است). فرم دارد (ایده و نقد). نمای فهرست با فیلتر و جستجو دارد. دادهٔ رابطه‌ای دارد (کاربر، ایده، نقد). از نظر مفهومی به‌اندازهٔ کافی ساده است که بتوانیم روی معماری تمرکز کنیم بدون اینکه در پیچیدگی دامنه گم شویم.

این یک **MVP** (**Minimum Viable Product**) است. ویژگی‌های اصلی را می‌سازیم، اما طوری معماری می‌کنیم که رشد کند. بهبودهای آینده می‌تواند شامل همکاری روی ایده‌ها، رأی‌گیری، اعلان‌ها، پیشنهادهای مبتنی بر AI، analytics و موارد دیگر باشد. معماری خوب یعنی آن ویژگی‌ها بدون بازساخت همه‌چیز اضافه شوند.

**درک نیازمندی‌ها**

قبل از تصمیم‌گیری معماری باید بفهمیم چه می‌سازیم. نیازمندی‌ها دو نوع هستند:

### نیازمندی‌های کارکردی: چه کاری انجام می‌دهد {#h2_34}

این‌ها ویژگی‌ها را از منظر کاربر توصیف می‌کنند. تعریف روشن نیازمندی‌های کارکردی قبل از نوشتن کد دو هدف دارد. اول، نیازمندی‌های مشخصی برای تصمیم‌گیری معماری به ما می‌دهند؛ ابزار را به‌صورت انتزاعی انتخاب نمی‌کنیم، ابزاری انتخاب می‌کنیم که با آنچه باید بسازیم بخواند.

دوم، معیار پذیرش می‌شوند تا بدانیم یک ویژگی چه زمانی تمام شده است. هنگام خواندن این‌ها، فکر کنید هر نیازمندی به کدام یک از چالش‌های معماری که قبلاً بحث کردیم مربوط است.

- **احراز هویت و مدیریت کاربر:** بنیاد یک پلتفرم اجتماعی هویت است. کاربران باید بتوانند حساب بسازند، وارد شوند و حضور خود را مدیریت کنند:
  - **ثبت‌نام کاربر:** کاربران جدید می‌توانند با نام کاربری، ایمیل و رمز عبور حساب بسازند
  - **ورود/خروج کاربر:** کاربران ثبت‌نام‌شده می‌توانند احراز هویت کنند و session خود را مدیریت کنند
  - **پروفایل کاربر:** کاربران می‌توانند پروفایل خود را ببینند و ویرایش کنند، از جمله اطلاعات bio با پشتیبانی از قالب‌بندی Markdown
  - **داشبورد کاربر:** کاربران احراز هویت‌شده یک داشبورد شخصی‌سازی‌شده برای مدیریت محتوای خود دریافت می‌کنند
- **مدیریت ایده:** هستهٔ پلتفرم خود ایده‌هاست. کاربران به کنترل کامل روی ایجاد، مرور و مدیریت ایده‌های اپلیکیشن AI نیاز دارند:
  - **ایجاد ایده:** کاربران واردشده می‌توانند ایدهٔ جدید اپلیکیشن AI با عنوان، توضیح کوتاه، توضیح مفصل (با پشتیبانی Markdown) و tag ارسال کنند
  - **مرور ایده‌ها:** همهٔ بازدیدکنندگان می‌توانند ایده‌ها را در فهرست صفحه‌بندی‌شده مرتب‌شده بر اساس تاریخ ایجاد مرور کنند
  - **مشاهدهٔ جزئیات ایده:** نمای جزئیات با اطلاعات کامل ایده، شامل همهٔ نقدهای مرتبط
  - **ویرایش ایده‌ها:** کاربران می‌توانند ایده‌های خود را برای بهبود و به‌روزرسانی محتوا ویرایش کنند
  - **حذف ایده‌ها:** کاربران می‌توانند ایده‌های خود را حذف کنند (با حذف آبشاری نقدهای مرتبط)
  - **جستجو و فیلتر:** کاربران می‌توانند ایده‌ها را بر اساس عنوان یا توضیح جستجو کنند و بر اساس tag فیلتر کنند
- **سیستم نقد:** ارزش جامعه از بازخورد می‌آید. سیستم نقد به کاربران اجازه می‌دهد ایده‌های یکدیگر را از طریق سیستم نقد ارزیابی و دربارهٔ آن‌ها بحث کنند، از جمله:
  - **نوشتن نقد:** کاربران واردشده می‌توانند برای ایده‌ها نقد با امتیاز ۱ تا ۵ ستاره و محتوای متنی بنویسند
  - **مشاهدهٔ نقدها:** همهٔ بازدیدکنندگان می‌توانند نقدها را در صفحات جزئیات ایده ببینند، مرتب‌شده بر اساس تاریخ
  - **ویرایش نقدها:** کاربران می‌توانند نقدهای خود را به‌روزرسانی کنند
  - **حذف نقدها:** کاربران می‌توانند نقدهای خود را حذف کنند
  - **محدودیت‌های نقد:** هر کاربر فقط می‌تواند یک نقد برای هر ایده بنویسد
- **کشف محتوا:** پلتفرم فقط وقتی مفید است که کاربران بتوانند آنچه می‌خواهند پیدا کنند. ویژگی‌های کشف به نمایان کردن محتوای مرتبط کمک می‌کنند:
  - **فهرست ایده‌ها:** فهرست صفحه‌بندی‌شده یا infinite scroll همهٔ ایده‌ها
  - **فیلتر مبتنی بر tag:** فیلتر ایده‌ها با انتخاب tag
  - **جستجوی real-time:** قابلیت جستجو با به‌روزرسانی debounced
  - **نمای محتوای کاربر:** کاربران می‌توانند همهٔ ایده‌ها و نقدهای خود را در صفحات اختصاصی ببینند

### نیازمندی‌های غیرکارکردی: چگونه کار می‌کند {#h2_35}

این‌ها ویژگی‌های کیفی و محدودیت‌ها را توصیف می‌کنند — ویژگی‌هایی که تعیین می‌کنند اپلیکیشن حرفه‌ای و قابل اعتماد به نظر برسد، حتی وقتی چیز visible «خراب» نیست. نیازمندی‌های غیرکارکردی راحت نادیده گرفته می‌شوند اما برای موفقیت اپلیکیشن مهم‌اند. هر دستهٔ زیر استانداردی را تعریف می‌کند که در کل کتاب به آن پایبند خواهیم بود:

- **Performance:** اپلیکیشن باید استانداردهای performance وب مدرن را برآورده کند:
  - بارگذاری اولیهٔ صفحه ظرف ۳ ثانیه روی broadband معمولی
  - navigation سمت client باید instant به نظر برسد (< 100ms)
  - اندازهٔ bundle زیر ۱MB برای بارگذاری اولیهٔ صفحه
  - برآورده کردن آستانه‌های Core Web Vitals (LCP < 2.5s، FID < 100ms، CLS < 0.1)
- **Usability:** اپلیکیشن باید user-friendly و intuitive باشد:
  - responsive design که روی دستگاه‌ها از عرض ۳۲۰px تا ۱۹۲۰px کار کند
  - رابط touch-friendly برای دستگاه‌های موبایل
  - navigation یکسان در همهٔ routeها
  - loading state و skeleton screen واضح هنگام data fetching
  - form validation intuitive با پیام خطای مفید
- **Accessibility:** اپلیکیشن باید برای همهٔ کاربران accessible باشد:
  - برآورده کردن استانداردهای accessibility WCAG 2.1 AA
  - پشتیبانی keyboard navigation برای همهٔ elementهای interactive
  - ARIA label و semantic HTML درست
  - نسبت contrast رنگ حداقل 4.5:1 برای متن
- **Security:** measureهای امنیتی client-side باید پیاده‌سازی شوند:
  - input validation و sanitization برای جلوگیری از حملات XSS
  - sanitization محتوای Markdown قبل از render
  - ذخیرهٔ امن token authentication (`httpOnly` cookies)
  - محافظت در برابر آسیب‌پذیری‌های رایج وب
- **Reliability:** اپلیکیشن باید failureها را gracefully handle کند:
  - network error handling با مکانیزم retry
  - React error boundary برای handle خطای component
  - graceful degradation وقتی feature در دسترس نیست
  - fallback content مناسب برای data fetching ناموفق

### مدل داده {#h2_36}

اپلیکیشن ما سه entity اصلی دارد که با هم پلتفرم اجتماعی را می‌سازند:

- **User**
  - `id`: شناسهٔ یکتا
  - `username`: نام کاربری یکتا
  - `email`: آدرس email کاربر
  - `password`: password hash شده برای authentication
  - `bio`: biography کاربر با پشتیبانی Markdown (اختیاری)
  - `createdAt`: timestamp ایجاد حساب
- **Idea**
  - `id`: شناسهٔ یکتا
  - `title`: عنوان ایده
  - `shortDescription`: خلاصهٔ کوتاه ایده
  - `description`: توضیح مفصل با پشتیبانی Markdown
  - `tags`: آرایهٔ tag برای categorization
  - `authorId`: reference به user سازنده
  - `createdAt`: timestamp ایجاد
  - `updatedAt`: timestamp آخرین به‌روزرسانی
- **Review**
  - `id`: شناسهٔ یکتا
  - `content`: متن نقد
  - `rating`: امتیاز ستاره‌ای (۱–۵)
  - `ideaId`: reference به idea مورد نقد
  - `authorId`: reference به User نویسندهٔ نقد
  - `createdAt`: timestamp ایجاد
  - `updatedAt`: timestamp آخرین به‌روزرسانی

روابط بین این entityها:

- یک User می‌تواند many Idea بسازد (one-to-many)
- یک User می‌تواند many Review بنویسد (one-to-many)
- یک Idea می‌تواند many Review داشته باشد (one-to-many)
- هر Review متعلق به یک Idea و یک User است
- هر User فقط یک Review per Idea می‌نویسد (constraint enforced)

این یک مدل relational تمیز است. همهٔ نیازهای MVP را پوشش می‌دهد و به‌اندازهٔ کافی ساده است که سریع فهمیده شود.

#### تصمیم‌های معماری {#h3_37}

حالا می‌توانیم تصمیم آگاهانه بگیریم. تعریف کردیم اپ چه می‌کند (functional requirement)، چقدر خوب باید انجام دهد (non-functional requirement) و با چه داده‌ای کار می‌کند (data model). هر تصمیم معماری زیر پاسخ مستقیم به این requirementهاست.

ابزار را به‌خاطر popularity انتخاب نمی‌کنیم؛ به‌خاطر حل مسائل specific اپلیکیشنمان. برای هر تصمیم approach را نام می‌بریم و توضیح می‌دهیم چرا با requirement ما می‌خواند.

#### ساختار پروژه: مبتنی بر feature {#h3_38}

بر اساس domain feature (users، ideas، reviews) سازمان‌دهی می‌کنیم نه نقش فنی. هر پوشهٔ feature شامل component، API call، hook، type و utility خودش است. code مشترک (auth، UI component، utility) در پوشه‌های shared است.

با رشد اپ scale می‌شود. featureها مستقل develop می‌شوند. code مرتبط co-located است. تازه‌وارد با فهم featureها ساختار را می‌فهمد.

#### استراتژی رندر: hybrid {#h3_39}

صفحات مختلف نیاز متفاوت دارند. از React Router v7 استفاده می‌کنیم که موارد زیر را پشتیبانی می‌کند:

- **Static Site Generation (SSG)** برای landing page: pre-render در build time، load instant، عالی برای marketing
- **Server-Side Rendering (SSR)** برای صفحات public idea: render on-demand روی server، خوب برای SEO و initial load performance
- **Client-Side Rendering (CSR)** برای صفحات authenticated: interactivity غنی، navigation سریع بعد از initial load

همچنین code splitting برای lazy load route، prefetching برای resource critical، TanStack Query برای cache پاسخ API، و skeleton screen برای loading state.

React Router v7 این را straightforward می‌کند. hybrid rendering approach یعنی هر route rendering method خودش را specify کند.

#### مدیریت state: چندلایه {#h3_40}

انواع مختلف state ابزار مختلف می‌خواهند:

- Local state (component-specific): `useState` و `useReducer`
- Global application state (در context اپ ما فقط سیستم notification): Zustand
- Server state (دادهٔ API): TanStack Query
- Form state: React Hook Form با validation Zod
- URL state (search، filter، pagination): کتابخانهٔ Nuqs — [https://nuqs.dev/](https://nuqs.dev/)

این over-engineering را اجتناب می‌کند. هر ابزار برای use case خودش ساخته شده.

#### استایل: Tailwind CSS + BaseUI {#h3_41}

Tailwind CSS برای utility-first styling. سریع است (build-time نه runtime)، consistency را با design token enforce می‌کند و purge feature bundle را کوچک نگه می‌دارد. developer experience عالی است وقتی utilityها را یاد بگیرید.

برای component primitive (dropdown، dialog و...) از Base UI استفاده می‌کنیم. Base UI componentهای unstyled و accessible می‌دهد. keyboard navigation، focus management و ARIA attribute را handle می‌کند. با Tailwind استایل می‌دهیم.

کنترل کامل design با accessibility عالی built-in.

#### احراز هویت: مبتنی بر Cookie {#h3_42}

`httpOnly` cookie برای authentication. cookie توسط server set می‌شود و خودکار با request فرستاده می‌شود. JavaScript دسترسی ندارد — محافظت در برابر XSS.

در frontend، Zustand user فعلی را track می‌کند. protected route کاربر unauthenticated را به login redirect می‌کند. auth error را gracefully handle می‌کنیم و session را با check با server بعد از refresh حفظ می‌کنیم.

#### تست: pragmatic {#h3_43}

testing trophy را دنبال می‌کنیم: بیشتر integration test، بعضی E2E برای مسیر critical، unit test برای logic پیچیده.

- **Unit tests (Vitest)**: utility function، custom hook، محاسبات
- **Integration tests (Playwright + Mocked API)**: تست صفحهٔ کامل از منظر کاربر
- **End-to-end tests (Playwright)**: journeyهای critical (register، login، create idea، write review)

presentational component یا integration third-party را over-test نمی‌کنیم. هدف confidence است نه coverage percentage.

## خلاصه {#h1_44}

زمین زیادی پوشش دادیم. دربارهٔ اهمیت معماری حرف زدیم — نه فقط در theory بلکه در practice. landscape معماری اپلیکیشن‌های React و تصمیم‌هایی که با آن روبه‌رو می‌شوید را explore کردیم. patternهای مؤثر و ناکارآمد را دیدیم. و اپ real برنامه‌ریزی کردیم که با هم می‌سازیم.

تصمیم‌هایی که گرفتیم arbitrary نیستند. driven by requirementهای ما هستند. اپ دیگر با requirement دیگر ممکن است انتخاب متفاوت بخواهد. نکته همین است. معماری دربارهٔ follow کردن rule نیست. دربارهٔ فهم trade-off و تصمیم آگاهانه است.

فصل بعد development environment و tooling را setup می‌کنیم. همهٔ چیز لازم برای ساخت این اپلیکیشن را install و configure می‌کنیم. سپس piece by piece می‌سازیم و می‌بینیم این تصمیم‌های معماری در practice چگونه عمل می‌کنند.

هدف فقط ساخت اپلیکیشن نیست. فهمیدن *why* است که چرا این‌طور می‌سازیم. آن فهم است که در پروژه‌های خودتان با requirement و context خودتان تصمیم معماری خوب می‌گیرید.

آماده‌اید؟ قرار نیست فقط اپ React بسازیم. بیایید یکی *architect* کنیم.
