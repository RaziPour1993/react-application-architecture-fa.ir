# فصل ۱: درک معماری اپلیکیشن‌های React

React محبوب‌ترین کتابخانهٔ JavaScript در جهان برای ساخت رابط کاربری است. بازار کار پر از فرصت‌های مبتنی بر React است.

React به دلایل متعددی این‌قدر محبوب شده است: API کوچکش، اکوسیستم عظیم کامپوننت‌های آماده‌اش، و انعطاف‌پذیری‌اش.

اما برای توسعه‌دهندهٔ امروزی، همین انعطاف می‌تواند شمشیر دو لبه باشد. React نظر قطعی دربارهٔ نحوهٔ ساختاردهی اپلیکیشن نمی‌دهد؛ بنابراین هر تیمی باید خودشان راه خودشان را پیدا کنند. این امر به ناسازگاری بین پروژه‌ها، تصمیم‌های معماری تحت فشار deadline بدون درک کامل trade-offها، و codebaseهایی منجر می‌شود که با رشد، نگهداری‌شان سخت‌تر و سخت‌تر می‌شود.

این یعنی هر تیمی با سؤالات مهمی روبه‌روست: اپلیکیشن را چگونه ساختاربندی کنیم؟ state چگونه جریان می‌یابد؟ routing، استایل کامپوننت‌ها و هر تصمیم معماری دیگر را چگونه بگیریم؟ React پاسخ قطعی نمی‌دهد.

با این حال، تصمیم‌های معماری خوب تفاوت بین ساخت یک اپلیکیشن قابل نگهداری و ایجاد هرج‌ومرجی از spaghetti code سخت‌دیباگ است.

هدف این کتاب همان شفافیتی است که جا افتاده: چگونه اپلیکیشن React را طوری معماری کنیم که مقیاس‌پذیر، قابل نگهداری و همراه با رشد شما و تیم‌تان باشد.

در این فصل موارد زیر را پوشش می‌دهیم:

- بررسی مزایای داشتن معماری خوب اپلیکیشن
- بررسی چالش‌های معماری اپلیکیشن‌های React
- درک تصمیم‌های معماری هنگام ساخت اپلیکیشن React
- برنامه‌ریزی اپلیکیشنی که در این کتاب می‌سازیم

در پایان این فصل، خواهید دانست چگونه از منظر معماری به اپلیکیشن‌های React نگاه کنید و تصمیم‌های آگاهانه‌ای بگیرید که موفقیت بلندمدت پروژه را تضمین می‌کند.

## چرا معماری مهم است {#h1_15}

هر اپلیکیشنی معماری دارد — حتی آن‌هایی که کسی به آن فکر نکرده است. ممکن است پروژهٔ React‌ای باز کنید و ۴۷ کامپوننت در یک پوشهٔ `components/`، فراخوانی‌های API پراکنده در فایل‌های تصادفی، و state‌ای ببینید که بسته به نویسندهٔ کد به سه روش مختلف مدیریت شده است.

این هم معماری است. فقط عمدی نبوده.

سؤال این نیست که اپلیکیشن شما معماری دارد یا نه؛ سؤال این است که آن معماری عمدی انتخاب شده یا تصادفی شکل گرفته است.

وقتی از معماری حرف می‌زنیم، منظورمان تصمیم‌های سطح بالایی است که شکل کار اپلیکیشن را تعیین می‌کنند. کد چگونه سازمان‌دهی شده؟ بخش‌های مختلف چگونه با هم ارتباط دارند؟ تیم چه الگوهایی را دنبال می‌کند؟ این تصمیم‌ها بعداً سخت عوض می‌شوند چون همهٔ چیزهایی که روی آن‌ها ساخته شده تحت تأثیر قرار می‌گیرند.

در این فصل بررسی می‌کنیم چرا معماری اصلاً مهم است، چالش‌های خاص React چیست، الگوهای مؤثر و ناکارآمد کدام‌اند، و اپلیکیشن واقعی‌ای که در ادامهٔ کتاب می‌سازیم را برنامه‌ریزی می‌کنیم.

### معماری به‌عنوان بنیان {#h2_16}

وقتی ساختمانی می‌سازید، از پی شروع می‌کنید. اول دیوار را نمی‌سازید و بعد پی را در نظر نمی‌گیرید. نرم‌افزار هم همین‌طور است، اما پیامدها فوری نیستند.

می‌توانید بدون فکر به معماری feature بسازید و مدتی همه‌چیز خوب پیش برود. مشکل وقتی پدیدار می‌شود که بخواهید چیزی را عوض کنید.

بنیان معماری محکم، اپلیکیشن را در برابر تغییر مقاوم می‌کند. و تغییر تنها ثابت نرم‌افزار است. نیازمندی‌ها عوض می‌شوند. feature جدید اضافه می‌شود. اعضای تیم می‌آیند و می‌روند. فناوری‌های جدید ظاهر می‌شوند. اولویت‌های کسب‌وکار می‌چرخدند. اپلیکیشنی که روی بنیان محکم ساخته شده می‌تواند با این تغییرات سازگار شود. اپلیکیشنی که روی بنیان لرزان ساخته شده به workaroundهای گران‌تر و گران‌تر نیاز دارد تا در نهایت به گران‌ترین گزینه برسد: بازنویسی کامل.

### هماهنگی و سرعت توسعه {#h2_17}

معماری خوب کار تیمی را آسان‌تر می‌کند. وقتی کامپوننت‌ها مرز و مسئولیت روشنی دارند، توسعه‌دهندگان می‌توانند روی بخش‌های مختلف اپلیکیشن بدون تداخل با یکدیگر کار کنند. می‌توانید کار را بهتر بین اعضای تیم تقسیم کنید. می‌توانید تخمین دقیق‌تری بزنید چون scope تغییرات را می‌فهمید.

این موضوع با بزرگ شدن تیم اهمیت بیشتری پیدا می‌کند. یک توسعه‌دهندهٔ تنها می‌تواند کل اپلیکیشن بد معماری را در ذهنش نگه دارد، اما این روش scale نمی‌شود. وقتی نفر دوم می‌پیوندد باید الگوها را بفهمد. وقتی نفر سوم می‌پیوندد باید الگوها یکسان باشند. وقتی پنج نفر شدید، معماری ناهمگون اصطکاک دائمی ایجاد می‌کند.

معماری بد روی توسعه با کمک AI هم اثر می‌گذارد. وقتی از ابزارهایی مثل GitHub Copilot یا Claude استفاده می‌کنید، آن‌ها بر اساس دادهٔ آموزشی خود و الگوهای codebase شما پیشنهاد می‌دهند. معماری یکدست یعنی پیشنهادهای یکدست. معماری ناهمگون یعنی AI نمی‌داند کدام الگو را دنبال کند و زمان بیشتری صرف اصلاح پیشنهادهایش می‌شود تا آنچه از استفادهٔ آن صرفه‌جویی کرده‌اید.

### سرعت از طریق شفافیت {#h2_18}

برخلاف تصور اولیه، وقت گذاشتن روی معماری از ابتدا در واقع سرعت توسعه را بالا می‌برد. وقتی تصمیم‌های معماری گرفته شده باشند، توسعه‌دهندگان می‌توانند روی حل مسائل کسب‌وکار تمرکز کنند نه بحث فنی برای هر feature. feature جدید در الگوهای موجود جا می‌گیرد. تازه‌واردها سریع‌تر productive می‌شوند چون ساختار روشن است.

جایگزین، چیزی است که می‌توانیم *architecture decision fatigue* بنامیم. هر feature جدید نیاز به تصمیم‌گیری دارد: چگونه کار کند، کجا قرار بگیرد، چگونه با بخش‌های دیگر ارتباط بگیرد. این micro-decisionها جمع می‌شوند، ناسازگاری ایجاد می‌کنند و همه‌چیز را کند می‌کنند.

این موضوع هنگام کار با LLM برای code generation به‌ویژه مهم است. اگر از AI بخواهید feature‌ای به اپلیکیشن خوب معماری‌شده اضافه کند، می‌تواند الگوهای موجود را دنبال کند. اگر codebase ناهمگون باشد، باید حدس بزند کدام الگو را دنبال کند — و اغلب اشتباه حدس می‌زند.

### اقتصاد معماری خوب {#h2_19}

پروژه‌های نرم‌افزاری هزینه دارند. بیشتر این هزینه مربوط به افراد است: زمان، تلاش و توجه آن‌ها. معماری خوب با مؤثرتر کردن افراد هزینه‌ها را کاهش می‌دهد. زمان debug کردن کمتر می‌شود. زمان جستجو برای یافتن چیزها کمتر می‌شود. زمان دور زدن محدودیت‌های معماری کمتر می‌شود.

معماری ضعیف هزینهٔ تصاعدی دارد. هر feature جدید زمان بیشتری می‌برد چون باید دور مشکلات موجود کار کنید. هر باگ سخت‌تر رفع می‌شود چون مرز بین componentها نامشخص است. در نهایت به نقطه‌ای می‌رسید که افزودن feature هزینهٔ سنگینی دارد و بازنویسی ضروری می‌شود. بازنویسی‌ها گران و پرریسک هستند.

معماری خوب در تصمیم‌گیری‌های کسب‌وکار هم کمک می‌کند. وقتی ساختار اپلیکیشن را می‌فهمید، هزینه‌ها را دقیق‌تر تخمین می‌زنید. می‌توانید تصمیم بگیرید کدام feature را بسازید و کدام را به بعد موکول کنید.

### معماری و کیفیت محصول {#h2_20}

وقتی توسعه‌دهندگان کمتر با معماری دست‌وپنجه نرم می‌کنند، وقت بیشتری برای تمرکز روی نیازهای کاربر دارند. می‌توانند روی polish کار کنند، دربارهٔ edge case فکر کنند، error handling را بهبود دهند و اپلیکیشن را accessible‌تر کنند.

معماری ضعیف چرخهٔ معیوبی ایجاد می‌کند. توسعه‌دهندگان وقتشان صرف رفع مشکلات معماری می‌شود. featureها با لبه‌های خشن عرضه می‌شوند. کاربران باگ می‌بینند. توسعه‌دهندگان وقت بیشتری صرف رفع اضطراری مشکلات می‌کنند. هیچ‌وقت برای بهبود معماری وقت نیست چون همیشه چیز فوری‌ای وجود دارد.

معماری خوب چرخهٔ مثبت بهبود کیفیت ایجاد می‌کند. توسعه‌دهندگان وقتشان صرف ساخت feature می‌شود. featureها با کیفیت بهتر عرضه می‌شوند. کاربران راضی‌ترند. باگ کمتری گزارش می‌شود. تیم وقت دارد معماری را سالم نگه دارد.

هدف هر نرم‌افزار حل مسئلهٔ واقعی برای انسان‌های واقعی است. همهٔ چیزهای دیگر در خدمت همان هدف است. معماری مهم است چون تعیین می‌کند روی همان هدف بمانید یا در technical debt غرق شوید.

## چالش معماری در React {#h1_21}

React ابزار عالی‌ای برای ساخت رابط کاربری است، اما عمداً بعضی مسائل را حل نمی‌کند. فهمیدن اینکه React چه می‌کند و چه نمی‌کند، به درک تصمیم‌های معماری لازم کمک می‌کند.

انعطاف React هم برکت است و هم نفرین. برکت است چون کتابخانه شما را مجبور نمی‌کند الگویی را بپذیرید که با مسئله‌تان نمی‌خواند. نفرین است چون مسئولیت هر تصمیمی با خود شماست. این موضوع به محبوبیت React کمک کرده و اکوسیستم غنی‌ای از کتابخانه‌ها و ابزارها به وجود آورده — که می‌توانید آن‌ها را تصمیم‌های از پیش گرفته‌شده بدانید. اما این خود مشکل جدیدی ایجاد کرده: *choice overload*.

### چگونه پروژه را ساختاربندی کنیم؟ {#h2_22}

React نظری دربارهٔ file structure ندارد. می‌توانید:

- همه‌چیز را در یک فایل بگذارید (هرچند نباید این کار را بکنید!)
- پوشهٔ ساده‌ای از componentها داشته باشید
- بر اساس نقش فنی سازمان‌دهی کنید (همهٔ componentها در یک پوشه، همهٔ hookها در پوشهٔ دیگر)
- بر اساس feature سازمان‌دهی کنید
- بر اساس مفهوم domain سازمان‌دهی کنید

Dan Abramov، یکی از اعضای سابق تیم مرکزی React، جملهٔ معروفی دارد: *«فایل‌ها را آن‌قدر جابه‌جا کن تا حس درستی پیدا کنی.»* این جمله برای بعضی‌ها راهگشاست و برای بعضی‌ها ناامیدکننده. راهگشاست چون می‌پذیرد اپلیکیشن‌های مختلف نیازهای متفاوتی دارند. شبکهٔ اجتماعی و text editor لزوماً نباید یک‌سان سازمان‌دهی شوند. ناامیدکننده است چون نقطهٔ شروعی به شما نمی‌دهد.

ساختار مهم‌تر از آن است که فکر می‌کنید. ساختار خوب یافتن را آسان می‌کند. ساختار بد یعنی انرژی ذهنی صرف navigation در codebase می‌شود. وقتی LLM را روی codebase خود آموزش می‌دهید یا از ابزارهای AI pair programming استفاده می‌کنید، ساختار بخشی از context window می‌شود. ساختار روشن به AI کمک می‌کند اپلیکیشن شما را بفهمد.

### چگونه اپلیکیشن را رندر کنیم؟ {#h2_23}

اصطلاح *rendering* در React چندین معنا دارد که می‌تواند گیج‌کننده باشد. وقتی از استراتژی rendering در React حرف می‌زنیم، منظور این است: کد کجا و کی اجرا می‌شود تا UI مشخص شود؟

ساده‌ترین گزینه **client-side rendering (CSR)** است: همهٔ کد React در مرورگر اجرا می‌شود. سرور فایل HTML حداقلی می‌فرستد، JavaScript دانلود و اجرا می‌شود و React UI را می‌سازد. این روش برای اپلیکیشن‌های پشت authentication یا ابزارهای داخلی عالی است.

اما اگر بخواهید crawl برای موتورهای جستجو تا حد ممکن آسان باشد؟ آن وقت شاید به **server-side rendering (SSR)** نیاز داشته باشید: کد React روی سرور اجرا می‌شود و HTML تولید می‌کند که به مرورگر فرستاده می‌شود. مرورگر آن را فوراً نشان می‌دهد (برای perceived performance خوب است)، سپس JavaScript را دانلود می‌کند و صفحه را hydrate می‌کند تا interactive شود.

یا شاید محتوای شما زیاد عوض نمی‌شود و بتوانید از **static site generation (SSG)** استفاده کنید: HTML در build time از پیش رندر شود و از CDN سرو شود.

یا شاید به هر سه روش برای بخش‌های مختلف اپلیکیشن نیاز داشته باشید. React و frameworkهای مختلف از همهٔ این‌ها پشتیبانی می‌کنند، اما باید تصمیم بگیرید کجا از کدام روش استفاده کنید.

React Server Components بُعد دیگری به این سؤال اضافه می‌کند. بعضی componentها فقط روی سرور اجرا می‌شوند و کدشان هرگز به client فرستاده نمی‌شود. گاهی این موضوع به bundle size کمک می‌کند، اما قوانین جدیدی دربارهٔ آنچه آن componentها می‌توانند و نمی‌توانند انجام دهند ایجاد می‌کند و ممکن است دادهٔ بیشتری در HTML بفرستد نسبت به آنچه در bundle صرفه‌جویی می‌شود.

### چگونه state را مدیریت کنیم؟ {#h2_24}

React هک‌های **`useState`** و **`useReducer`** برای state محلی و Context API برای اشتراک‌گذاری state بین componentها فراهم می‌کند. برای بسیاری از اپلیکیشن‌ها این کافی است. اما برای بسیاری دیگر کافی نیست.

چالش این است که *state* یک چیز واحد نیست. state محلی UI وجود دارد (dropdown باز است؟)، state فرم (کاربر چه تایپ کرده؟)، state سراسری اپلیکیشن (کاربر فعلی کیست؟)، server state (چه داده‌ای از API گرفته‌ایم؟) و URL state (روی چه صفحه‌ای هستیم، search parameterها چیست؟).

هر کدام ویژگی‌ها و نیازهای متفاوتی دارند. state محلی UI معمولاً ساده است. server state پیچیده است چون caching، invalidation، background update و error handling دارد. یکسان دیدن همهٔ stateها به مشکل ختم می‌شود.

اکوسیستم راه‌حل‌های زیادی دارد: Redux، MobX، Zustand، Jotai، Recoil، XState و موارد دیگر. سپس ابزارهای تخصصی برای server state مثل TanStack Query (React Query) و SWR. هر کدام فلسفه و trade-off متفاوتی دارند.

### چگونه کامپوننت‌ها را استایل دهیم؟ {#h2_25}

بعضی توسعه‌دهندگان فایل‌های CSS سنتی را ترجیح می‌دهند. بعضی CSS Modules برای scoping. بعضی utility-first CSS مثل Tailwind. بعضی CSS-in-JS و موارد دیگر.

این انتخاب فقط به سلیقهٔ developer بستگی ندارد. روی performance (build-time در برابر runtime)، نحوهٔ فکر کردن دربارهٔ reusability کامپوننت‌ها، consistency طراحی و bundle size تأثیر می‌گذارد.

### چگونه data fetching را انجام دهیم؟ {#h2_26}

React data fetching داخلی ندارد. در عوض چندین گزینه دارید:

- می‌توانید مستقیماً از `fetch` API مرورگر استفاده کنید
- می‌توانید از کتابخانه‌ای مثل [Axios](https://axios-http.com/) استفاده کنید
- می‌توانید از ابزار تخصصی مثل [TanStack Query](https://tanstack.com/query/latest) استفاده کنید که cache و background update را مدیریت می‌کند
- می‌توانید از قابلیت‌های data fetching یک meta-framework مثل [Next.js](https://nextjs.org/) استفاده کنید

*کجا* fetch کنید هم مهم است. در component؟ در custom hook؟ در لایهٔ data جداگانه؟ روی سرور؟ هر رویکردی روی نحوهٔ تست کد و مدیریت loading و error state تأثیر دارد.

### چگونه احراز هویت را پیاده‌سازی کنیم؟ {#h2_27}

احراز هویت با چند تصمیم دیگر تلاقی دارد. از token-based auth استفاده می‌کنید (توکن‌ها در `localStorage` یا `sessionStorage` ذخیره می‌شوند)؟ یا cookie-based auth (کوکی‌های `httpOnly` خودکار با request فرستاده می‌شوند)؟ cookie-based امن‌تر است اما نیاز به session handling سمت سرور دارد. token-based ساده‌تر است اما در برابر XSS آسیب‌پذیرتر است اگر درست مدیریت نشود.

مسیرها چگونه محافظت می‌شوند؟ خطاهای احراز هویت چگونه مدیریت می‌شوند؟ token refresh چگونه انجام می‌شود؟

این الگوها باید در کل اپلیکیشن یکسان باشند.

### چگونه اپلیکیشن React را تست کنیم؟ {#h2_28}

اپلیکیشن‌های React را می‌توان در چندین سطح تست کرد: تست توابع منفرد، تست componentها، تست end-to-end برای flow کامل کاربر و regression test برای تغییرات UI.

اما زمان بی‌نهایت نیست. کدام تست بیشترین اطمینان را با کمترین maintenance burden می‌دهد؟ implementation detail تست می‌کنید یا رفتار کاربر؟ componentهایی که data fetch می‌کنند چطور؟ flowهای احراز هویت‌شده چطور؟

استراتژی تست روی معماری تأثیر دارد، چون بعضی الگوهای معماری test-friendly‌تر از بقیه هستند.

### تصمیم متا {#h2_29}

زیر همهٔ این‌ها یک تصمیم متا وجود دارد: از meta-framework مثل Next.js، React Router یا TanStack Start استفاده می‌کنید؟

این frameworkها بعضی تصمیم‌های بالا را برایتان می‌گیرند، اما شما را به الگوهای خود lock می‌کنند.

لزوماً بد نیست. constraintها گاهی آزادتان می‌کنند تا کمتر به معماری فکر کنید و بیشتر روی خود اپلیکیشن تمرکز کنید.

اما این انتخابی است که همهٔ چیزهای دیگر را تحت تأثیر قرار می‌دهد.

## اندیشیدن دربارهٔ تصمیم‌های معماری {#h1_30}

همهٔ تصمیم‌های معماری برابر نیستند. بعضی تقریباً در هر شرایطی clearly wrong هستند. بعضی clearly right. بیشتر جایی بین این دو قرار دارند که پاسخ درست به شرایط خاص شما بستگی دارد. بیایید چند مثال ببینیم.

### الگوهایی که معمولاً جواب نمی‌دهند {#h2_31}

بعضی الگوهای معماری در نگاه اول معقول به نظر می‌رسند اما با رشد اپلیکیشن مشکل‌ساز می‌شوند. چند نمونه از codebaseهای React و دلیل شکست آن‌ها:

- **Everything in One Place**: گذاشتن همهٔ componentها در یک پوشهٔ ساده وقتی ۱۰ component دارید خوب کار می‌کند. با ۵۰ component پیدا کردن چیزها کار طاقت‌فرسایی می‌شود. با ۱۰۰ component فاجعه است. همین‌طور دربارهٔ گذاشتن کل اپلیکیشن در یک فایل صدق می‌کند. اینکه *می‌توانید* کاری را انجام دهید به معنای این نیست که *باید* این کار را بکنید.

  چرا جواب نمی‌دهد؟ چون انسان‌ها با لیست‌های بلند و بی‌تمایز خوب کار نمی‌کنند. برای فهمیدن چیزها به ساختار نیاز داریم. فایل‌سیستم بخشی از مستندات اپلیکیشن است. باید ساختار را نشان دهد نه پنهان کند.

- **One Big Component**: componentی که همه‌کاره است در ابتدا efficient به نظر می‌رسد. همهٔ logic در یک جاست. اما این componentها غیرقابل نگهداری می‌شوند. تست کردنشان سخت است چون باید کل محیط اطراف component را راه‌اندازی کنید. استفادهٔ مجدد از آن‌ها سخت است چون به چیزهای زیادی وابسته‌اند. درک‌شان سخت است چون باید چیز زیادی را همزمان در ذهن نگه دارید.

- **Large components**: وقتی از AI برای تغییر کد استفاده می‌کنید، componentهای بزرگ مشکل‌ساز می‌شوند. component ممکن است بخش زیادی از context window AI را پر کند، یا AI تغییراتی پیشنهاد دهد که تعاملات ظریف داخل component را در نظر نگیرد.

- **Everything global**: گذاشتن همهٔ state در store سراسری ساده به نظر می‌رسد. همه‌چیز در یک جا! اما در واقع کار را سخت‌تر می‌کند. وقتی هر componentی می‌تواند هر stateای را تغییر دهد، فهم data flow غیرممکن می‌شود. نمی‌توانید به component نگاه کنید و بفهمید به چه stateای وابسته است. تغییرات پرریسک می‌شوند چون نمی‌دانید چه چیز دیگری ممکن است بشکند. state سراسری جای خودش را دارد، اما بیشتر state باید local باشد. state را تا جایی که ممکن است نزدیک محل استفاده نگه دارید و فقط وقتی share کنید که واقعاً لازم است.

- **Wrong Tool, Right Problem**: اکوسیستم React ابزارهای تخصصی برای مسائل تخصصی دارد. استفاده از state manager عمومی برای مدیریت server data (دادهٔ fetch شده از API) یعنی reimplement کردن cache، revalidation، optimistic update و background sync. این‌ها مسائل پیچیده‌ای هستند. ابزارهایی مثل TanStack Query آن‌ها را حل کرده‌اند و در scale اثبات شده‌اند. نادیده گرفتن ابزار تخصصی برای اجتناب از dependency add کردن pragmatic به نظر می‌رسد، اما در نهایت به کد بیشتر، باگ بیشتر و maintenance بیشتر منجر می‌شود. یک dependency به‌دقت انتخاب‌شده راه‌حل است، نه مشکل.

- **Ignoring Security**: sanitize نکردن ورودی کاربر یک تصمیم معماری نیست — این یک اشتباه است. اگر اپلیکیشن شما محتوای user-generated نشان می‌دهد (و ما یکی از آن‌ها را می‌سازیم)، باید آن را sanitize کنید. وگرنه در برابر حملاتی آسیب‌پذیرید که در آن کاربران مخرب script تزریق می‌کنند تا در مرورگر کاربران دیگر اجرا شود. این موضوع به‌ویژه اگر از Markdown یا rich text استفاده می‌کنید مهم است.

- **Ignoring Performance Infrastructure**: سرو اپلیکیشن از یک سرور در یک موقعیت جغرافیایی به این معناست که کاربران دور از آن سرور load time کندی را تجربه می‌کنند. این مسئله حل شده است: CDNها اپلیکیشن را به‌صورت جهانی توزیع می‌کنند. کاربران از نزدیک‌ترین edge location سرو می‌شوند. این برای اپلیکیشن‌های وب مدرن حداقل انتظار است.

### الگوهایی که معمولاً جواب می‌دهند {#h2_32}

از طرف دیگر، الگوهایی وجود دارند که در تیم‌ها و codebaseهای هر اندازه‌ای خود را ثابت کرده‌اند. این‌ها rule نیستند؛ تصمیم‌هایی هستند که مگر دلیل خوبی برای انحراف داشته باشید، معمولاً به کارتان می‌آیند:

- **Feature-Based Structure**: سازمان‌دهی کد بر اساس feature نه نقش فنی بهتر scale می‌شود. به‌جای پوشه برای همهٔ componentها، پوشه برای همهٔ hookها و پوشه برای همهٔ API callها، پوشه‌ای برای user authentication، پوشه‌ای برای ideas و پوشه‌ای برای reviews دارید. هر پوشه همهٔ چیزی که به آن feature مربوط است را در خود جای داده و تیم‌ها با merge conflict کمتر روی featureهای جداگانه کار می‌کنند. چرا جواب می‌دهد؟ چون featureها همان شکلی هستند که ما به اپلیکیشن فکر می‌کنیم. وقتی باید نحوهٔ کار ideas را عوض کنید، به پوشهٔ ideas می‌روید. همهٔ چیز لازم در یک جاست. تازه‌وارد با فهمیدن featureها، اپلیکیشن را می‌فهمد. این موضوع برای AI-assisted development هم مفید است: وقتی LLM در حال تغییر یک feature است، context مرتبط همه در یک جاست و AI راحت‌تر می‌تواند دامنهٔ کامل کار را درک کند.

- **Co-located State**: state را تا جایی که ممکن است local نگه دارید. اگر فقط یک component به آن نیاز دارد، از `useState` در همان component استفاده کنید. اگر چند component نزدیک به آن نیاز دارند، آن را به parent مشترک lift کنید. فقط اگر واقعاً نیاز سراسری است (احراز هویت کاربر، theme preference و...) آن را در store سراسری بگذارید. این کار scope state را روشن نگه می‌دارد. وقتی به component نگاه می‌کنید، می‌بینید به چه stateای وابسته است. تغییرات کمتر risky هستند چون impact محدود و مشخص است.

- **Appropriate Specialization**: از ابزار تخصصی برای مسائل تخصصی استفاده کنید. TanStack Query برای server state. React Hook Form برای فرم‌های پیچیده. Zod برای validation. این ابزارها وجود دارند چون مسائل سخت را خوب حل می‌کنند. کد مخصوص اپلیکیشن باید روی مسائل مخصوص اپلیکیشن تمرکز کند نه reimplement کردن راه‌حل‌های generic.

- **Layered Defense**: امنیت یک تصمیم نیست — مجموعه‌ای از تصمیم‌هاست. از کوکی‌های `httpOnly` برای token احراز هویت استفاده کنید تا JavaScript نتواند به آن‌ها دسترسی پیدا کند. ورودی را هم در سمت client و هم در سمت سرور اعتبارسنجی کنید. محتوای user-generated را قبل از render sanitization کنید. از TypeScript برای گرفتن خطاهای type در compile time استفاده کنید. از ESLint برای گرفتن اشتباهات رایج استفاده کنید. هر لایه چیزهایی را می‌گیرد که لایه‌های دیگر از دست می‌دهند.

- **Small, Focused Components**: componentها باید یک کار انجام دهند. اگر componentی بزرگ می‌شود، نشانهٔ نیاز به شکستن آن است. componentهای کوچک‌تر understand، test، modify و reuse کردنشان آسان‌تر است. برای AI هم راحت‌تر هستند چون در context window جا می‌شوند و رفتارشان روشن‌تر است.

- **Tooling and Analysis**: از ابزارها برای enforce کردن consistency استفاده کنید — ESLint برای code quality، Prettier برای formatting و TypeScript برای type safety. این ابزارها خطا را قبل از اجرای کد می‌گیرند. cognitive load را کم می‌کنند چون ruleها را به خاطر نمی‌سپارید. consistency بین تیم ایجاد می‌کنند. این موضوع برای AI code generation به‌ویژه ارزشمند است: وقتی پیشنهادهای AI خودکار format و type-check می‌شوند، مشکلات فوراً دیده می‌شوند.

  این الگوها و ضدالگوها منویی از امکانات معماری به ما می‌دهند. اما الگو به‌تنهایی فقط تا اینجا پیش می‌برد. برای دیدن عملی این ایده‌ها به اپلیکیشنی واقعی با نیازمندی‌های واقعی نیاز داریم. همان را برنامه‌ریزی می‌کنیم.

## برنامه‌ریزی اپلیکیشن {#h1_33}

بحث انتزاعی دربارهٔ معماری مفید است، اما تصمیم‌های معماری در بستر اپلیکیشن واقعی با نیازمندی‌های واقعی گرفته می‌شوند. پس یکی می‌سازیم.

در این کتاب **AI Ideas Community Platform** می‌سازیم — اپلیکیشنی اجتماعی که در آن کاربران ایده‌های اپلیکیشن‌های AI را به اشتراک می‌گذارند، ایده‌های دیگران را مرور می‌کنند و برایشان نقد می‌نویسند. این یک مرکز جامعه برای کسانی است که می‌خواهند آنچه در فضای AI می‌سازند — یا فکر می‌کنند بسازند — را با دیگران به اشتراک بگذارند.

چرا این اپلیکیشن؟ چون بیشتر تصمیم‌های معماری که بحث کردیم را لمس می‌کند. صفحات عمومی دارد (برای SEO). صفحات احراز هویت‌شده دارد (برای ارسال ایده و نقد). محتوای تولیدشده توسط کاربر دارد (یعنی sanitization لازم است). فرم دارد (ایده و نقد). نمای فهرست با فیلتر و جستجو دارد. دادهٔ رابطه‌ای دارد (کاربر، ایده، نقد). از نظر مفهومی به‌اندازهٔ کافی ساده است که بتوانیم روی معماری تمرکز کنیم بدون اینکه در پیچیدگی دامنه گم شویم.

این یک **MVP** (**Minimum Viable Product**) است. ویژگی‌های اصلی را می‌سازیم، اما طوری معماری می‌کنیم که قابل رشد باشد. بهبودهای آینده می‌تواند شامل همکاری روی ایده‌ها، رأی‌گیری، اعلان‌ها، پیشنهادهای مبتنی بر AI، analytics و موارد دیگر باشد. معماری خوب یعنی آن ویژگی‌ها بدون بازساختن همه‌چیز اضافه شوند.

**درک نیازمندی‌ها**

قبل از تصمیم‌گیری معماری باید بفهمیم چه می‌سازیم. نیازمندی‌ها دو نوع هستند:

### نیازمندی‌های کارکردی: چه کاری انجام می‌دهد {#h2_34}

این‌ها ویژگی‌ها را از منظر کاربر توصیف می‌کنند. تعریف روشن نیازمندی‌های کارکردی قبل از نوشتن کد دو هدف دارد. اول، نیازمندی‌های مشخصی برای تصمیم‌گیری معماری به ما می‌دهند؛ ابزار را به‌صورت انتزاعی انتخاب نمی‌کنیم، ابزاری انتخاب می‌کنیم که با آنچه باید بسازیم همخوانی داشته باشد.

دوم، این نیازمندی‌ها معیار پذیرش می‌شوند تا بدانیم یک ویژگی چه زمانی تمام شده است. هنگام خواندن این‌ها، فکر کنید هر نیازمندی به کدام یک از چالش‌های معماری که قبلاً بحث کردیم مربوط است.

- **احراز هویت و مدیریت کاربر:** بنیاد یک پلتفرم اجتماعی هویت است. کاربران باید بتوانند حساب بسازند، وارد شوند و حضور خود را مدیریت کنند:
  - **ثبت‌نام کاربر:** کاربران جدید می‌توانند با نام کاربری، ایمیل و رمز عبور حساب بسازند
  - **ورود/خروج کاربر:** کاربران ثبت‌نام‌شده می‌توانند احراز هویت کنند و session خود را مدیریت کنند
  - **پروفایل کاربر:** کاربران می‌توانند پروفایل خود را ببینند و ویرایش کنند، از جمله اطلاعات bio با پشتیبانی از قالب‌بندی Markdown
  - **داشبورد کاربر:** کاربران احراز هویت‌شده یک داشبورد شخصی‌سازی‌شده برای مدیریت محتوای خود دریافت می‌کنند
- **مدیریت ایده:** هستهٔ پلتفرم خود ایده‌هاست. کاربران به کنترل کامل روی ایجاد، مرور و مدیریت ایده‌های اپلیکیشن‌های AI نیاز دارند:
  - **ایجاد ایده:** کاربران واردشده می‌توانند ایدهٔ جدید اپلیکیشن AI با عنوان، توضیح کوتاه، توضیح مفصل (با پشتیبانی Markdown) و tag ارسال کنند
  - **مرور ایده‌ها:** همهٔ بازدیدکنندگان می‌توانند ایده‌ها را در فهرست صفحه‌بندی‌شده مرتب‌شده بر اساس تاریخ ایجاد مرور کنند
  - **مشاهدهٔ جزئیات ایده:** نمای جزئیات با اطلاعات کامل ایده، شامل همهٔ نقدهای مرتبط
  - **ویرایش ایده‌ها:** کاربران می‌توانند ایده‌های خود را برای بهبود و به‌روزرسانی محتوا ویرایش کنند
  - **حذف ایده‌ها:** کاربران می‌توانند ایده‌های خود را حذف کنند (با حذف آبشاری نقدهای مرتبط)
  - **جستجو و فیلتر:** کاربران می‌توانند ایده‌ها را بر اساس عنوان یا توضیح جستجو کنند و بر اساس tag فیلتر کنند
- **سیستم نقد:** ارزش جامعه از بازخورد می‌آید. سیستم نقد به کاربران اجازه می‌دهد ایده‌های یکدیگر را از طریق سیستم نقد ارزیابی و دربارهٔ آن‌ها بحث کنند، از جمله:
  - **نوشتن نقد:** کاربران واردشده می‌توانند برای ایده‌ها نقد با امتیاز ۱ تا ۵ ستاره و محتوای متنی بنویسند
  - **مشاهدهٔ نقدها:** همهٔ بازدیدکنندگان می‌توانند نقدها را در صفحات جزئیات ایده ببینند، مرتب‌شده بر اساس تاریخ
  - **ویرایش نقدها:** کاربران می‌توانند نقدهای خود را به‌روزرسانی کنند
  - **حذف نقدها:** کاربران می‌توانند نقدهای خود را حذف کنند
  - **محدودیت‌های نقد:** هر کاربر فقط می‌تواند یک نقد برای هر ایده بنویسد
- **کشف محتوا:** پلتفرم فقط وقتی مفید است که کاربران بتوانند آنچه می‌خواهند پیدا کنند. ویژگی‌های کشف به نمایان کردن محتوای مرتبط کمک می‌کنند:
  - **فهرست ایده‌ها:** فهرست صفحه‌بندی‌شده یا infinite scroll همهٔ ایده‌ها
  - **فیلتر مبتنی بر tag:** فیلتر ایده‌ها با انتخاب tag
  - **جستجوی real-time:** قابلیت جستجو با به‌روزرسانی debounced
  - **نمای محتوای کاربر:** کاربران می‌توانند همهٔ ایده‌ها و نقدهای خود را در صفحات اختصاصی ببینند

### نیازمندی‌های غیرکارکردی: چگونه کار می‌کند {#h2_35}

این‌ها ویژگی‌های کیفی و محدودیت‌ها را توصیف می‌کنند — ویژگی‌هایی که تعیین می‌کنند اپلیکیشن حرفه‌ای و قابل اعتماد به نظر برسد، حتی وقتی چیزی به‌صورت ظاهری «خراب» نیست. نیازمندی‌های غیرکارکردی راحت نادیده گرفته می‌شوند اما برای موفقیت اپلیکیشن بسیار مهم هستند. هر دستهٔ زیر استانداردی را تعریف می‌کند که در کل کتاب به آن پایبند خواهیم بود:

- **Performance:** اپلیکیشن باید استانداردهای performance وب مدرن را برآورده کند:
  - بارگذاری اولیهٔ صفحه ظرف ۳ ثانیه روی broadband معمولی
  - navigation سمت client باید instant به نظر برسد (< 100ms)
  - اندازهٔ bundle زیر ۱MB برای بارگذاری اولیهٔ صفحه
  - برآورده کردن آستانه‌های Core Web Vitals (LCP < 2.5s، FID < 100ms، CLS < 0.1)
- **Usability:** اپلیکیشن باید کاربرپسند و intuitiv باشد:
  - responsive design که روی دستگاه‌هایی با عرض ۳۲۰px تا ۱۹۲۰px کار کند
  - رابط touch-friendly برای دستگاه‌های موبایل
  - navigation یکسان در همهٔ مسیرها
  - loading state و skeleton screen واضح هنگام data fetching
  - form validation intuitiv با پیام‌های خطای مفید
- **Accessibility:** اپلیکیشن باید برای همهٔ کاربران accessible باشد:
  - برآورده کردن استانداردهای accessibility WCAG 2.1 AA
  - پشتیبانی از keyboard navigation برای همهٔ elementهای interactive
  - ARIA label و semantic HTML درست
  - نسبت contrast رنگ حداقل ۴.۵:۱ برای متن
- **Security:** اقدامات امنیتی سمت client باید پیاده‌سازی شوند:
  - input validation و sanitization برای جلوگیری از حملات XSS
  - sanitization محتوای Markdown قبل از render
  - ذخیرهٔ امن token احراز هویت (`httpOnly` cookies)
  - محافظت در برابر آسیب‌پذیری‌های رایج وب
- **Reliability:** اپلیکیشن باید failureها را به‌صورت graceful مدیریت کند:
  - مدیریت خطای network با مکانیزم retry
  - React error boundary برای مدیریت خطای component
  - degradation gracefully وقتی feature در دسترس نیست
  - fallback content مناسب برای data fetching ناموفق

### مدل داده {#h2_36}

اپلیکیشن ما سه entity اصلی دارد که با هم پلتفرم اجتماعی را می‌سازند:

- **User**
  - `id`: شناسهٔ یکتا
  - `username`: نام کاربری یکتا
  - `email`: آدرس ایمیل کاربر
  - `password`: رمز عبور hash شده برای احراز هویت
  - `bio`: biography کاربر با پشتیبانی Markdown (اختیاری)
  - `createdAt`: timestamp ایجاد حساب
- **Idea**
  - `id`: شناسهٔ یکتا
  - `title`: عنوان ایده
  - `shortDescription`: خلاصهٔ کوتاه ایده
  - `description`: توضیح مفصل با پشتیبانی Markdown
  - `tags`: آرایهٔ tag برای categorization
  - `authorId`: ارجاع به کاربر سازنده
  - `createdAt`: timestamp ایجاد
  - `updatedAt`: timestamp آخرین به‌روزرسانی
- **Review**
  - `id`: شناسهٔ یکتا
  - `content`: متن نقد
  - `rating`: امتیاز ستاره‌ای (۱–۵)
  - `ideaId`: ارجاع به idea مورد نقد
  - `authorId`: ارجاع به کاربر نویسندهٔ نقد
  - `createdAt`: timestamp ایجاد
  - `updatedAt`: timestamp آخرین به‌روزرسانی

روابط بین این entityها:

- یک User می‌تواند بسیاری Idea بسازد (one-to-many)
- یک User می‌تواند بسیاری Review بنویسد (one-to-many)
- یک Idea می‌تواند بسیاری Review داشته باشد (one-to-many)
- هر Review متعلق به یک Idea و یک User است
- هر User فقط یک Review برای هر Idea می‌نویسد (محدودیت اجرایی)

این یک مدل رابطه‌ای تمیز است. همهٔ نیازهای MVP را پوشش می‌دهد و به‌اندازهٔ کافی ساده است که سریع فهمیده شود.

#### تصمیم‌های معماری {#h3_37}

حالا می‌توانیم تصمیم‌های آگاهانه بگیریم. تعریف کردیم اپلیکیشن چه می‌کند (نیازمندی کارکردی)، چقدر خوب باید انجام دهد (نیازمندی غیرکارکردی) و با چه داده‌ای کار می‌کند (مدل داده). هر تصمیم معماری زیر پاسخ مستقیم به این نیازمندی‌هاست.

ابزار را به‌خاطر popularity انتخاب نمی‌کنیم؛ به‌خاطر حل مسائل مشخص اپلیکیشنمان. برای هر تصمیم رویکرد را نام می‌بریم و توضیح می‌دهیم چرا با نیازمندی ما همخوانی دارد.

#### ساختار پروژه: مبتنی بر feature {#h3_38}

بر اساس featureهای domain (users، ideas، reviews) سازمان‌دهی می‌کنیم نه نقش فنی. هر پوشهٔ feature شامل component، API call، hook، type و utility خودش است. کد مشترک (auth، UI component، utility) در پوشه‌های shared قرار می‌گیرد.

با رشد اپلیکیشن scale می‌شود. featureها مستقل develop می‌شوند. کد مرتبط co-located است. تازه‌وارد با فهمیدن featureها ساختار را می‌فهمد.

#### استراتژی رندر: hybrid {#h3_39}

صفحات مختلف نیازهای متفاوتی دارند. از React Router v7 استفاده می‌کنیم که موارد زیر را پشتیبانی می‌کند:

- **Static Site Generation (SSG)** برای landing page: pre-render در build time، load instant، عالی برای marketing
- **Server-Side Rendering (SSR)** برای صفحات عمومی idea: render on-demand روی سرور، خوب برای SEO و initial load performance
- **Client-Side Rendering (CSR)** برای صفحات احراز هویت‌شده: interactivity غنی، navigation سریع بعد از initial load

همچنین از code splitting برای lazy load کردن مسیرها، prefetching برای منابع حیاتی، TanStack Query برای cache کردن پاسخ‌های API و skeleton screen برای loading state استفاده خواهیم کرد.

React Router v7 این کار را ساده می‌کند. رویکرد hybrid rendering یعنی هر مسیر rendering method خودش را مشخص کند.

#### مدیریت state: چندلایه {#h3_40}

انواع مختلف state ابزارهای مختلفی می‌خواهند:

- Local state (مخصوص component): `useState` و `useReducer`
- Global application state (در اپلیکیشن ما فقط سیستم notification): Zustand
- Server state (دادهٔ API): TanStack Query
- Form state: React Hook Form با validation Zod
- URL state (search، filter، pagination): کتابخانهٔ [Nuqs](https://nuqs.dev/)

این رویکرد از over-engineering جلوگیری می‌کند. هر ابزار برای use case خودش ساخته شده است.

#### استایل: Tailwind CSS + BaseUI {#h3_41}

Tailwind CSS برای utility-first styling. سریع است (build-time نه runtime)، consistency را با design token enforce می‌کند و purge feature bundle را کوچک نگه می‌دارد. developer experience عالی است وقتی utilityها را یاد بگیرید.

برای component primitives (dropdown، dialog و...) از Base UI استفاده می‌کنیم. Base UI componentهای unstyled و accessible ارائه می‌دهد. keyboard navigation، focus management و ARIA attributes را مدیریت می‌کند. با Tailwind استایلشان می‌دهیم.

این کنترل کامل طراحی با accessibility عالی built-in فراهم می‌کند.

#### احراز هویت: مبتنی بر Cookie {#h3_42}

از `httpOnly` cookie برای احراز هویت استفاده می‌کنیم. کوکی توسط سرور set می‌شود و خودکار با request فرستاده می‌شود. JavaScript دسترسی به آن ندارد — محافظت در برابر XSS.

در frontend، Zustand کاربر فعلی را رهگیری می‌کند. مسیرهای محافظت‌شده کاربر unauthenticated را به login redirect می‌کنند. خطاهای احراز هویت را به‌صورت graceful مدیریت می‌کنیم و session را با بررسی با سرور بعد از refresh حفظ می‌کنیم.

#### تست: pragmatic {#h3_43}

testing trophy را دنبال می‌کنیم: بیشتر integration test، بعضی E2E برای مسیرهای حیاتی و unit test برای logic پیچیده.

- **Unit tests (Vitest)**: utility function، custom hook، محاسبات
- **Integration tests (Playwright + Mocked API)**: تست صفحهٔ کامل از منظر کاربر
- **End-to-end tests (Playwright)**: journeyهای حیاتی (register، login، ایجاد idea، نوشتن نقد)

presentational component یا integration third-party را over-test نمی‌کنیم. هدف confidence است نه coverage percentage.

## خلاصه {#h1_44}

زمین زیادی را پوشش دادیم. دربارهٔ اهمیت معماری حرف زدیم — نه فقط در theory بلکه در practice. landscape معماری اپلیکیشن‌های React و تصمیم‌هایی که با آن‌ها روبه‌رو می‌شوید را بررسی کردیم. الگوهای مؤثر و ناکارآمد را دیدیم. و اپلیکیشن واقعی را برنامه‌ریزی کردیم که با هم می‌سازیم.

تصمیم‌هایی که گرفتیم arbitrary نیستند. بر اساس نیازمندی‌های ما هستند. اپلیکیشن دیگری با نیازمندی‌های دیگر ممکن است انتخاب‌های متفاوتی بخواهد. نکته همین است. معماری دربارهٔ دنبال کردن rule نیست. دربارهٔ فهمیدن trade-offها و تصمیم‌گیری آگاهانه است.

در فصل بعد development environment و tooling را راه‌اندازی می‌کنیم. همهٔ چیزی که برای ساخت این اپلیکیشن لازم است را install و configure می‌کنیم. سپس piece by piece می‌سازیم و می‌بینیم این تصمیم‌های معماری در practice چگونه عمل می‌کنند.

هدف فقط ساخت اپلیکیشن نیست. فهمیدن *why* است — چرا این‌طور می‌سازیم. آن فهم است که به شما امکان می‌دهد در پروژه‌های خودتان، با نیازمندی‌ها و شرایط خودتان، تصمیم‌های معماری خوب بگیرید.

آماده‌اید؟ قرار نیست فقط اپلیکیشن React بسازیم. بیایید یکی *architect* کنیم.
